## 9단계 — 사용자 로그 수집 서버 (배포 이후)

> 조건: 실사용자 배포 완료 후. 보안 설계 선행.

### 방향 검토

**옵션 A — 로컬/포트 연결:**
- 장점: 구현 단순, 비용 없음
- 단점: NAT·방화벽 설정 필요, TLS 없으면 평문 전송, 공인 IP 없으면 외부 접근 불가

**옵션 B — 클라우드 (권장):**
- 후보: Supabase (PostgreSQL, 무료 500MB/5만 req/월), PlanetScale (MySQL), Cloudflare D1 (SQLite)
- 장점: HTTPS 기본, 인증 내장, 별도 서버 관리 불필요
- 단점: 무료 플랜 용량·요청 제한

### 구현 항목

| 항목 | 내용 |
|---|---|
| LOG-1 | 로그 스키마 설계 (`session_id`, `character_id`, `turn`, `user_input`, `assistant_response`, `category`, `importance`, `reviewed`, `logged_at`) |
| LOG-2 | 로컬 `training/log/` → 클라우드 DB 동기화 스크립트 |
| LOG-3 | 업로드 전 사용자 식별 정보 마스킹 |
| LOG-4 | 클라우드 DB → 로컬 학습 데이터 추출 파이프라인 (`build_sft_from_feedback.py` 확장) |
| LOG-5 | 로그 수집 동의 UI (첫 실행 시 opt-in 팝업) |

## 반려 / 영구 보류 항목

| 항목 | 사유 |
|---|---|
| 웹 검색 기능 | RAM/VRAM 절감 목적, 2026-04-06 삭제 완료 |
| llama-server HTTP 모드 | 배포 복잡도 증가, 오버엔지니어링 |
| ctypes 직접 호출 | 유지보수 부담 과도 |
| Qwen2.5-3B-Korean 교체 | 학습량 비대칭, 낮은 성능 평가 |
| summarizer 2-LLM-call 방식 | 오버스펙 — session_context는 LLM 불필요 |
| 절전 모드 | RAM 실제 문제 발생 시점에 재검토 |
| 경량 임베딩 모델 교체 | 현 BAAI/bge-m3 대비 리스크 있음 |
