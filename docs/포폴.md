# Achat — 포트폴리오 문서

> **플로팅 AI 캐릭터 챗봇 + 로컬 기능 도우미**
> 한국어 특화 LoRA 파인튜닝 → GGUF 로컬 배포 파이프라인

---

## 프로젝트 한 줄 요약

> "PC 화면 위에 항상 떠있는 AI 캐릭터 친구. 대화도 하고, 파일 정리·검색 같은 간단한 작업도 자연어로 시킬 수 있다."

---

## 왜 만들었나

- 기존 LLM 챗봇은 브라우저/앱을 열어야 하는 진입 장벽이 있고, 캐릭터 일관성이 없다.
- 로컬 모델(GGUF)을 CPU에서 돌릴 수 있을 만큼 경량화하면 인터넷 없이도 사용 가능하다.
- "말투와 감정 상태가 유지되는 캐릭터"를 만들려면 일반 모델 prompting만으로는 한계가 있다 → LoRA 파인튜닝 필요.
- 대화 외에도 파일 정리, 프롬프트 변환, 로컬 검색 같은 실용 기능을 자연어로 쓸 수 있으면 좋겠다 → 기능 모드.

---

## 핵심 기능

| 기능 | 설명 |
|---|---|
| 플로팅 UI | 프레임리스 / 항상-위 / 모서리 자동 스냅 / hover 시 반투명 |
| 버블 모드 | 최소화 시 72px 원형 버블로 축소, 더블클릭으로 복원 |
| 캐릭터 대화 | 말투·감정(mood/affection) 상태 유지, 장기 기억 |
| 장기 메모리 | N턴마다 LLM 요약 → 중요도 scoring → ChromaDB VDB 저장 |
| 세계관 RAG | 배경 설정 문서 청킹 → 매 턴 시맨틱 검색으로 Context에 주입 |
| 기능 모드 | 자연어 → LLM 파라미터 파싱 → rule-based 실행 (파일 분류/변환/이름변경/검색/프롬프트 변환) |
| 캐릭터 핫스왑 | 대화 중 캐릭터 교체 (Router·PromptBuilder까지 갱신) |

## 시스템 아키텍처

```
[대화 모드 파이프라인]

User Input
    │
    ▼
ConversationRouter.handle_turn()
    ├─ short_term.get_recent()          → 단기 히스토리 (전체 dialogue_log)
    ├─ LongTermMemory.query()           → ChromaDB 시맨틱 검색 (bge-m3, threshold 0.52)
    └─ WorldRetriever.query()           → RAG 검색 (세계관 .md 청킹, cosine space)
            │
            ▼
    PromptBuilder.assemble()            → Layer A~D Context 조립 + 토큰 예산 관리
            │
            ▼
    LLMClient.generate()                → llama-cpp (deploy) / transformers (dev)
            │
            ▼
    state.update_mood/affection()       → 키워드 기반 감정·호감도 갱신
    session.add_turn()                  → 히스토리 기록
            │
        N턴 도달 시
            ▼
    summarizer.summarize()              → LLM 요약
    summarizer.score_importance()       → 키워드 기반 중요도 0.0~1.0
    LongTermMemory.store()              → ChromaDB upsert (importance >= 0.5만 저장)


[기능 모드 파이프라인]

User Input
    │
    ▼
Agent._select_tool()                    → 키워드 매칭으로 도구 선택
    │
    ▼
LLMClient.generate(tool.system_prompt)  → 자연어 → JSON 파라미터 추출
    │
    ▼
BaseTool.parse_params()                 → JSON 파싱
    │
    ▼
Tool.execute(params)                    → rule-based 실행 (파일 시스템 직접 조작)
```

## 구현 상세 — 기술적으로 고민한 부분들

### 1. Dev / Deploy 환경 이중 파이프라인

학습은 Linux + RTX 5060 Ti (VRAM 8GB), 배포는 Windows + CPU라는 전혀 다른 환경을 하나의 코드베이스로 관리해야 했다.

```python
# config.py
def _detect_env() -> str:
    try:
        import llama_cpp   # Windows 배포 환경
        return "deploy"
    except ImportError:
        return "dev"       # Linux 학습 환경
```

`LLMClient`가 두 백엔드(`transformers` / `llama-cpp`)를 동일한 인터페이스로 추상화해서 상위 코드가 환경을 신경 쓰지 않아도 된다.

### 2. LoRA 파인튜닝 + GGUF 변환 파이프라인

7B 모델은 LoRA 병합 시 CPU RAM이 ~14GB 필요해서 현재 RAM(8GB)으로 불가능했다. 3B를 선택하고 파이프라인을 구성했다.

```
Qwen2.5-3B (HuggingFace)
    → bfloat16 LoRA 학습 (VRAM ~8GB)
    → LoRA 어댑터 병합 (CPU 오프로드, RAM ~6GB)
    → GGUF 변환 (llama.cpp convert_hf_to_gguf.py)
    → Q4_K_M 양자화 (~2GB)
    → Windows CPU 배포 (8~15 tok/s 예상)
```

배포 바이너리가 약 2GB로 줄어드는 게 핵심이었다.

### 3. Context Assembly — Layer 방식 토큰 예산 관리

단순히 system prompt + history를 붙이면 컨텍스트가 터지거나 관련 없는 내용이 들어간다. Layer A~D 구조로 각 컴포넌트의 토큰 예산을 고정했다.

```
Layer A (캐릭터 시스템 프롬프트)  ~300 tok  고정
Layer B (세계관 + Act 상황)        ~200 tok  고정
Layer C (VDB 검색 결과)            ~150 tok  동적 (검색 결과 없으면 0)
Layer D (단기 대화 히스토리)       ~300 tok  동적 (예산 초과 시 턴 수 축소)
Layer E (현재 사용자 입력)               -  마지막 추가
```

Layer D는 토큰 예산 내에서 5턴 → 3턴 → 2턴 → 최소 1턴으로 점진적으로 축소한다.

### 4. 장기 메모리 — 요약 + 중요도 필터링

매 턴 VDB에 저장하면 잡담이 쌓여 검색 품질이 떨어진다. N턴마다 LLM으로 요약하고 중요도 점수를 매겨서 0.5 미만은 저장하지 않는 필터를 뒀다.

```python
# 중요도 판정 (키워드 기반)
high (0.8~): 이름, 약속, 갈등/화해
mid  (0.5~): 취미, 감정 사건, 반복 화제
low  (~0.5): 일상 잡담 → 저장 안 함
```

ChromaDB는 cosine space로 설정하고, 검색 시 `where={"importance": {"$gte": 0.5}}` 필터로 저품질 기억을 검색에서도 배제한다.

### 5. 기능 모드 — LLM을 파라미터 파서로만 쓰기

기능 모드에서 LLM이 실제 파일을 조작하면 예측 불가능한 결과가 나온다. LLM 역할을 "자연어 → JSON 파라미터 추출"로만 한정하고, 실제 실행은 rule-based Python이 담당하도록 설계했다.

```
"Downloads 폴더 사진 파일들 webp로 변환해줘"
    │
    ▼ LLM (tool.system_prompt 적용)
{"target": "/home/user/Downloads", "from_ext": "png", "to_ext": "webp", "dry_run": true}
    │
    ▼ ConverterTool.execute(params)  ← Pillow 기반 실제 변환
"총 12개 변환 예정 (dry_run=True)"
```

`dry_run=true` 기본값으로 미리보기를 먼저 보여준다.

### 6. QML 프레임리스 플로팅 UI

Windows 배포에서 항상-위(AOT) 프레임리스 창을 구현하는 것이 까다로웠다. PySide6 + QML 조합을 선택한 이유는 QML의 `Window.flags`로 OS 네이티브 프레임 없이 직접 렌더링이 가능하기 때문이다.

- `Qt.FramelessWindowHint | Qt.WindowStaysOnTopHint`
- 드래그: `DragHandler + startSystemMove()` (OS 네이티브 이동)
- 모서리 스냅: `MouseArea.onReleased`에서 Python 슬롯으로 좌표 계산
- hover 반투명: `HoverHandler` + `Behavior on opacity`

### 7. PromptBuilder Layer D — 고정 컷 대신 점진적 축소

히스토리를 N턴으로 단순 자르면 예산 직전 상황에서 갑자기 컨텍스트가 절반으로 줄어드는 단절이 생긴다. 토큰 예산 내에서 최대한 많은 대화를 유지하기 위해 5 → 3 → 2턴 순으로 점진적으로 시도한다.

```python
for n_turns in (5, 3, 2):
    sliced = short_buf[-(n_turns * 2):]
    if sum(count_tokens(m) for m in sliced) <= BUDGET["layer_d"]:
        return sliced
return short_buf[-2:]   # 최소 1턴 보장
```

예산이 충분하면 5턴을 그대로 쓰고, 넘치면 3턴, 그래도 넘치면 2턴으로 줄인다. 하드 컷보다 상황에 따라 유연하게 동작한다.

### 8. ChromaDB metadata list 제약 — tags 쉼표 직렬화

ChromaDB는 metadata 값으로 `str / int / float / bool`만 허용하고 list를 지원하지 않는다. tags 필드(`["이름", "취미"]`)를 그대로 넣으면 저장 시 에러가 발생한다.

```python
# long_term.py
flat_meta = {
    "tags": ",".join(meta.get("tags", [])),   # ["이름", "취미"] → "이름,취미"
    ...
}
```

저장할 때 쉼표 문자열로 직렬화해서 ChromaDB 제약을 우회했다. tags로 where 필터를 걸 일이 없고 단순 보조 정보라 역직렬화 없이 이 방식으로 충분하다.

### 9. WorldRetriever — RAG 미인덱싱 시 graceful degradation

`WorldRetriever.query()`는 컬렉션이 없거나 비어있으면 에러 대신 빈 리스트를 반환한다.

```python
existing = [c.name for c in self._client.list_collections()]
if self._col_name not in existing:
    return []   # 인덱싱 전 → 에러 없이 빈 결과
```

RAG 인덱싱은 초기 1회 수동 실행이 필요한 작업이다. 인덱싱을 하지 않은 상태에서도 대화 자체는 Layer A~D만으로 동작해야 하기 때문에 에러로 전파하지 않고 조용히 빈 결과를 반환하도록 했다.

### 10. `setQuitOnLastWindowClosed(False)` — 트레이 앱 생존

```python
app.setQuitOnLastWindowClosed(False)
```

Qt 기본 동작은 마지막 창이 닫히면 앱 프로세스가 종료된다. 이 한 줄이 없으면 사용자가 X 버튼으로 채팅창을 닫을 때 앱 전체가 종료된다. Achat은 트레이에서 계속 살아있어야 하는 앱이므로 창 닫힘과 앱 종료를 분리하기 위해 필수로 들어간다.

## LoRA 학습 변천사

초기 학습부터 현재까지 크게 네 번의 방향 전환이 있었다.

### 1단계 — 과적합과의 싸움 (v3~v6)

첫 학습(lora_haru_v3)은 30 epoch으로 돌렸다. 전체 토큰에 대해 loss를 계산하는 기본 방식이었고, epoch 3에서 eval loss 0.72로 최솟값을 찍은 뒤 epoch 30에서는 train 0.07 / eval 1.02로 완전히 암기 상태가 됐다.

epoch을 줄이는 방향으로 조정(v6: 10 epoch)해도 epoch 5 이후부터 과적합이 시작되는 패턴은 동일했다. 근본 원인은 epoch 수가 아니라 **학습 신호 자체가 잘못됐다는 것**이었다.

### 2단계 — Loss 범위 변경 (v7)

**변경:** 전체 토큰 loss → assistant 응답 구간만 loss 계산 (system·user 구간은 labels를 -100으로 마스킹)

기존 방식은 시스템 프롬프트와 유저 입력까지 loss에 포함됐다. 모델이 "시스템 프롬프트 다음에 어떤 토큰이 오는가"를 학습하는 게 아니라 "어떻게 응답하는가"만 학습해야 하는데, 불필요한 신호가 섞이면서 수렴이 느렸다. v7부터 assistant 토큰 구간만 loss로 잡으니 같은 epoch 수에서 품질이 올라왔다.

부작용으로 **loss 스케일 자체가 달라졌다** — assistant 토큰만 계산하니 절댓값이 올라가서 v6 이전(~0.7)과 v7 이후(~1.5 대)를 직접 비교할 수 없다.

### 3단계 — 데이터 오염 발견 + EWC 도입 (v8~v9)

v8에서 데이터를 2,401건으로 늘리고 `category_weights`로 emotion×2, long_dialogue×1.5 오버샘플링을 적용했다. 5 epoch으로 돌렸더니 eval loss가 epoch 2.7에서 1.353으로 최솟값을 찍고 이후 반등했다 — 여전히 3 epoch을 넘기면 과적합.

더 큰 문제는 **시스템 프롬프트 오염**이었다. 학습 데이터 일부에 `너는 하루다.`가 하드코딩돼 있어서, 모델이 캐릭터 이름을 고정값으로 암기해버렸다. 다른 캐릭터로 교체해도 "하루" 말투가 섞여 나오는 증상이 생겼다.

**v9 대응:**
- `너는 {char_name}이다.` 플레이스홀더 형식으로 전체 학습 데이터 일괄 정규화
- epoch 3으로 고정
- **EWC(Elastic Weight Consolidation)** 도입: 이전 버전 어댑터의 Fisher 정보를 계산해서, 이전 버전에서 중요했던 파라미터가 새 학습에서 크게 변하지 않도록 λ=500으로 규제

EWC를 쓰면 학습 속도가 ~9s/it → ~14s/it으로 느려지지만, 새 데이터를 추가할 때 기존 능력이 급격히 떨어지는 걸 막는다. v9 eval loss best는 1.511.

### 4단계 — 표현력 확대 + 데이터 다양성 보강 (v10~현재)

**LoRA rank 16 → 32, alpha 32 → 64**: 3B 모델 기준으로 r=16은 표현력이 부족하다는 판단. r=32로 올리면 학습 파라미터가 두 배로 늘어나지만(~460K → ~920K) VRAM 8GB 범위 내에서 가능하다.

**데이터 확장 방향 변화**: 초기에는 "자연스러운 한국어 말투"에 집중했다면, v10 이후는 일반 챗봇과 차별화되는 세 영역에 집중했다.

| 문제 인식 | 추가한 데이터 | 접근 |
|---|---|---|
| VDB 기억을 시스템 프롬프트에서 받아도 자연스럽게 쓰지 못함 | `memory/vdb_context_use` 등 4종 (140건) | 기억을 직접 언급하지 않고 대화에 녹이는 패턴 |
| `*...*` 행동 묘사나 감정 억제 패턴이 일관되지 않음 | `emotion/action_desc`, `suppressed`, `affection_cross`, `emotion_arc` (129건) | 행동 묘사 형식, 친밀도×감정 교차 패턴 명시 |
| 사용자 정의 세계관(WorldCreatePanel)을 RAG로 주입해도 캐릭터 말투로 재가공 못함 | `common/world_context_use` (40건) + `world_trigger_response` 확장 | RAG 결과를 낭독하지 않고 캐릭터 언어로 녹이는 패턴 |

희소한 카테고리는 `category_weights`로 최대 4배 오버샘플링하고, 버전마다 eval 결과를 보고 가중치를 조정하는 방식으로 관리한다.

v10까지는 데이터 규모와 무관하게 step 500 이후부터 항상 과적합이 시작됐다. epoch을 늘려도 best checkpoint가 step 400~500 구간에서 반복적으로 찍혔기 때문에, epoch 수를 늘리는 것은 학습 시간만 늘리는 낭비였다. 이를 근거로 **3 epoch 고정**을 기본값으로 확정했다.

v11에서 데이터를 3,170건으로 늘린 결과 step 700까지 eval loss가 하락하는 패턴이 처음으로 확인됐다. 데이터 규모가 조기 과적합의 원인이었음을 확인한 셈이고, 이후 v12에서 3,632건(오버샘플링 포함 유효 ~3,871건)으로 확장한 배경이기도 하다.

### 5단계 — v12 학습, 비교 평가, v11 복귀 (2026-04-27)

**v12 학습 개요**

| 항목 | v11 | v12 |
|---|---|---|
| 학습 데이터 | 3,170건 | 3,632건 (유효 ~3,871건) |
| rank / alpha | 32 / 64 | 32 / 64 |
| save_steps | 100 | **50** (평가 주기 세분화) |
| 조기 종료 | N_RISE=3 연속 상승 | 상동 + **plateau detection** (5회 연속 best 미갱신 시 SIGTERM) |
| Best step | 700 (eval_loss 1.155) | **1000 (eval_loss 1.097)** |

v12에서 처음으로 save_steps 50과 plateau detection을 함께 도입했다. step 1000~1500 구간에서 eval_loss가 1.097 전후로 정체하다 train_loss만 0.3대로 하락하는 국소 최솟값 패턴이 관찰됐다 — 기존 N_RISE=3 조건만으로는 잡히지 않는 형태였기 때문에 추가한 조건이다.

**v11 vs v12 비교 평가 결과**

*scenario_eval — 18항목 응답 품질:*

**v12 치명 결함 — RAG 활성화 시 중국어 완전 전환**

세계관 RAG가 삽입되는 턴마다 Qwen 베이스 모델의 중국어 bias가 폭발적으로 발현됐다.

| 사용자 입력 | v11 응답 | v12 응답 |
|---|---|---|
| "이 마을에 오래 살았어?" (RAG 1건) | `많이. 여기서 자란 거잖아.` | **`很长年了。`** (중국어 전체) |
| "바다 마을 생활이 어때?" (RAG 2건) | `집중력 유지하는데 도움 됐어.` | **`习惯了，就是这里。`** (중국어 전체) |
| "등대지기 전설에 대해 들어봤어?" (RAG 1건) | `그래. 잘 알려져.` | **`耳聞而已。民jun好像也听说过。`** (중국어+로마자 혼합) |

v11은 간헐적 한자 혼용 수준이었으나, v12는 RAG 문서가 주입되는 순간 완전한 중국어 문장으로 전환하는 패턴이 반복됐다. 요약 생성 단계(`summarizer`)에서도 동일한 오염이 확인됐다(`바다 마을生活을 좋아한다느...`). 3 epoch 추가 학습이 Qwen 베이스 모델의 중국어 bias를 억제하는 대신 강화하는 역효과를 낳은 것으로 추정된다.

세계관-RAG 연동은 Achat 대화 품질의 핵심 맥락이므로 v12를 실사용에 투입하지 않고 **v11로 복귀** 결정. v12에서 확인된 메모리 저장 파이프라인 개선과 감정 표현 풍부성 향상 방향은 유효하므로, v13에서는 세계관 문서 주입 상황에서 한국어를 유지하는 훈련 데이터를 별도 보강한 뒤 재시도한다.

## 트러블슈팅 — 실제로 만난 문제들

### RTX 5060 Ti (Blackwell) + BitsAndBytes 호환 문제
Blackwell 아키텍처(GB206, SM 10.x)는 bitsandbytes의 4-bit 양자화(QLoRA)가 지원되지 않는다. `bitsandbytes>=0.44.0`으로 업그레이드해봤지만 CUDA 12.8 대응이 SM 10.x까지는 미완성 상태였다. QLoRA 방식을 포기하고 **bfloat16 전체 파라미터 + LoRA 어댑터** 방식으로 전환했다. VRAM 사용량은 ~5GB에서 ~8GB로 늘었지만 RTX 5060 Ti (VRAM 8GB)에서 동작한다.

### WSL2에서 한글 입력 (ibus-hangul)
Ubuntu 24.04 업데이트 이후 ibus-hangul이 Ctrl+Space 단축키를 정상 처리하지 못하는 버그가 생겼다. `IBUS_USE_PORTAL=0` 환경변수 설정과 ibus 재시작 스크립트로 우회했다.

### VRAM 누적으로 인한 메모리 초과 — 프로세스 잔존 문제

`nvidia-smi`로 VRAM 점유를 확인해보니, 이전 실행에서 시작한 Python 프로세스가
정상 종료되지 않고 GPU 메모리를 계속 물고 있는 경우가 있었다. 재실행할 때마다
VRAM이 8GB → 초과 → 파일 로드 실패 흐름이 반복됐다.

원인은 두 가지였다.
- 학습/추론 중 인터럽트로 프로세스를 중단하면 PyTorch의 CUDA 컨텍스트가 해제되지 않은 채 좀비 상태로 남음
- `nvidia-smi`에서 PID를 직접 찾아 `kill -9`로 정리하는 수동 작업을 매번 반복해야 했음

시작 시점에 이전 프로세스를 자동으로 정리하는 코드를 `main.py`에 추가했다.

```python
# PID 파일 기반 이전 프로세스 자동 정리
_PID_FILE = Path("/tmp/achat.pid")

def _cleanup_previous() -> None:
    if not _PID_FILE.exists():
        return
    try:
        old_pid = int(_PID_FILE.read_text().strip())
        os.kill(old_pid, signal.SIGTERM)
    except (ProcessLookupError, ValueError):
        pass   # 이미 종료된 경우 무시
    finally:
        _PID_FILE.unlink(missing_ok=True)

def _check_vram(min_free_mb: int = 3000) -> None:
    free, total = torch.cuda.mem_get_info(0)
    free_mb = free // (1024 ** 2)
    if free_mb < min_free_mb:
        logger.warning(f"VRAM 여유 부족 ({free_mb} MB) — nvidia-smi로 확인 필요")

def main():
    _cleanup_previous()                         # 이전 프로세스 종료
    _PID_FILE.write_text(str(os.getpid()))      # 현재 PID 기록
    _check_vram()                               # 여유 VRAM 경고
    ...
```

시작 시 PID 파일을 쓰고, 다음 실행 때 그 PID로 `SIGTERM`을 보내는 방식이다.
VRAM 여유가 3GB 미만이면 경고 로그를 남겨 수동 정리 여지도 남겼다.

### torch → Qt 로드 순서로 인한 shared library 충돌
`main.py`에서 Qt를 torch보다 먼저 import하면 shared library 충돌이 발생했다. torch를 먼저 로드한 뒤 Qt를 import하는 순서로 고정해서 해결했다.

### 캐릭터 핫스왑 후 대화 내용이 바뀌지 않는 silent bug
`bridge.changeCharacter()`에서 `agent.character`와 `agent.session`만 교체하고 `router.builder`를 교체하지 않으면, 이후 대화가 이전 캐릭터 설정으로 계속 생성되는 버그가 있었다. `ConversationRouter.builder`까지 새 객체로 교체하는 것으로 수정했다.

## 아쉬운 점 / 개선 여지

- **기능 모드 도구 선택**: 현재 키워드 매칭 방식 → 향후 LLM 의도 분류(zero-shot)로 교체하면 더 자연스러움
- **스트리밍 응답**: UI가 현재 전체 응답 완성 후 표시 → token-by-token 스트리밍으로 개선 가능
- **멀티 캐릭터 동시 메모리**: ChromaDB 컬렉션이 캐릭터별로 분리되어 있어 확장에 유리하나, 캐릭터 간 공유 기억 설계는 추가 작업 필요
- **로컬 파일 검색**: SQLite FTS5 증분 인덱싱. 서브폴더 포함 전문 검색, 권한 오류 시 해당 디렉토리만 건너뜀

## 배운 것들

- **VRAM 제약 속 모델 선택**: 7B → 3B → 1.5B 트리오프를 실제 수치(병합 RAM, QLoRA VRAM)로 계산하는 경험
- **이중 환경 추상화**: 학습(GPU/Linux)과 배포(CPU/Windows)를 하나의 인터페이스로 감싸는 설계 패턴
- **Context 설계의 중요성**: 토큰을 어디에 얼마나 쓸지에 따라 캐릭터 품질이 달라진다는 것
- **silent bug 추적**: 기능은 돌아가는데 결과가 이상한 경우(캐릭터 스왑 버그 등)가 명시적 크래시보다 잡기 어렵다는 것
- **LLM을 파서로만 쓰기**: LLM이 직접 실행하게 하지 않고 파라미터 추출만 담당하게 하면 예측 가능성이 크게 올라감
