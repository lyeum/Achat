# 대화 품질 설계 문서

> 목표:
> 1. **이질감이 덜한** 자연스러운 한국어 대화
> 2. **10턴 수준의 맥락 유지** — 요약 기반 메모리로 흐름 보존
> 3. **페르소나별 가이드라인** — 캐릭터에 따라 응답 방식이 달라지는 폭넓은 대화

---

## 관련 디렉토리 구조

대화 품질에 직접 관여하는 파일들의 구조입니다.

```
Achat/
│
├─ conversation/                   # 대화 엔진 전체
│   ├─ core/
│   │   ├─ llm_client.py          # LLM 추론 인터페이스 (llama-cpp-python)
│   │   │                         #   - 스트리밍 콜백 (토큰 단위 출력)
│   │   │                         #   - n_ctx 설정 (컨텍스트 윈도우)
│   │   ├─ prompt_build.py        # Context Assembly + 토큰 예산 관리
│   │   │                         #   - Layer A~E 조립
│   │   │                         #   - 토큰 카운트 초과 시 Layer D 축소
│   │   ├─ router.py              # 턴 처리 오케스트레이터 + Post-processing
│   │   │                         #   - Pre-processing → 메모리 검색 → 컨텍스트 조립
│   │   │                         #   - LLM 호출 → 상태 업데이트 → 메모리 쓰기 트리거
│   │   └─ session.py             # 대화 세션 상태
│   │                             #   - mood, affection, turn_count
│   │                             #   - dialogue_log (단기 버퍼용)
│   │
│   ├─ loader/
│   │   ├─ character_load.py      # 캐릭터 YAML → dict 변환
│   │   ├─ memory_load.py         # 메모리 초기 데이터 로더
│   │   └─ world_load.py          # 세계관 YAML → dict 변환
│   │
│   ├─ character/                  # 캐릭터 설정 파일 (페르소나 정의)
│   │   ├─ character_schema.yaml  # ✅ 캐릭터 YAML 계약 (슬롯 = 학습 카테고리 어휘)
│   │   ├─ CH_Haru.yaml           # 하루 — speech/affection/emotion/personality 슬롯 기반
│   │   └─ CH_Seonjae.yaml        # 선재 — 따뜻하지만 표현 서툰 캐릭터
│   │
│   ├─ world/                      # 세계관 설정 파일 (Act 기반 상황 정의)
│   │   ├─ W_schema.json
│   │   └─ W_sea.yaml
│   │
│   └─ memory_act/                 # 메모리 스키마 및 초기 데이터
│       ├─ M_schema.json           # VDB 저장 단위 스키마 (확장)
│       ├─ M_default.json          # 캐릭터별 기본 기억
│       └─ M_instance.json         # 세션 인스턴스 기억
│
├─ memory/                         # 메모리 관리 레이어
│   ├─ short_term.py              # 단기 버퍼 — 최근 N턴 슬라이딩 윈도우
│   ├─ long_term.py               # 장기 VDB — ChromaDB 저장 / 시맨틱 검색
│   └─ summarizer.py              # 요약 + 중요도 scoring + 쓰기 트리거
│
├─ rag/                            # 세계관 RAG 파이프라인
│   ├─ index.py                   # 문서 청킹 + ChromaDB 인덱싱
│   ├─ retrieve.py                # 시맨틱 유사도 검색 (bge-m3)
│   └─ sources/world/             # 세계관 원본 문서
│
└─ agent/                          # 상위 오케스트레이터
    ├─ core.py                    # 전체 흐름 조율
    ├─ persona.py                 # 캐릭터 핫스왑
    ├─ state.py                   # mood / affection 전환 규칙
    └─ router.py                  # 시동어 / 명령어 분기
```

## 전체 처리 흐름

```
User Input
    │
    ▼
┌─────────────────────────────────────────────────┐
│ 1. 장소 이동 감지        (router._handle_location) │
│    - 이동 의도 감지 (LLM 기반)                  │
│    - YAML act 매칭 → session.act_id 갱신         │
│    - 미매칭 시 RAG/LLM으로 동적 장소 생성        │
│    - session.location_context 업데이트           │
└──────────────────────┬──────────────────────────┘
                       │
           ┌───────────┴───────────┐
           ▼                       ▼
┌──────────────────┐   ┌───────────────────────────┐
│ 2. 단기 메모리   │   │ 3. 장기 메모리 VDB 검색   │
│   short_term.py  │   │    long_term.py            │
│                  │   │                            │
│  session의       │   │  입력 텍스트를 bge-m3로    │
│  dialogue_log    │   │  임베딩 → ChromaDB 검색    │
│  전체 반환       │   │  유사도 ≥ 0.7인 결과만    │
│  (검색 없음)     │   │  top-2 반환                │
└────────┬─────────┘   └─────────────┬─────────────┘
         │                           │
         └─────────────┬─────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│ 4. Context Assembly      (prompt_build.py)      │
│                                                 │
│  Layer A [고정 ~300tok]                         │
│    캐릭터 시스템 프롬프트                        │
│    → 이름, 성격, 말투 규칙, 금지 사항           │
│                                                 │
│  Layer B [동적 ~200tok+]                        │
│    세계관 설명 + 현재 Act 상황                   │
│    + 세계관 RAG 결과 병합 (있을 때, 우선순위↓) │
│    → 장소, 분위기, 진행 중인 서사               │
│                                                 │
│  Layer C [동적 ~150tok]  ← 캐릭터 관점 재서술  │
│    장기 메모리 VDB 결과 (없으면 생략)           │
│    우선순위: VDB(장기메모리) > 세계관 RAG       │
│    → "바다를 좋아한다고 했었지." 형태로 변환    │
│                                                 │
│  Layer D [동적 ~300tok]  ← 초과 시 턴 수 축소  │
│    최근 3~5턴 대화 히스토리                     │
│                                                 │
│  Layer E [가변]                                 │
│    현재 사용자 입력                              │
│                                                 │
│  토큰 예산 합계: ~950tok + 생성 여유 512tok     │
│  → 총 ~1,462tok (3B 컨텍스트 내 여유 충분)     │
└──────────────────────┬──────────────────────────┘
                       │
                       ▼
              LLM 생성 (llama-cpp)
                       │
                       ▼
┌─────────────────────────────────────────────────┐
│ 5. Post-processing       (router.py)            │
│    - mood 변화 감지 (사용자 입력 기준)           │
│    - affection 증감 (mood 기반)                  │
│    - 메모리 쓰기 트리거 체크 (N턴 도달?)        │
└──────────────────────┬──────────────────────────┘
                       │
              N턴 조건 충족 시
                       ▼
┌─────────────────────────────────────────────────┐
│ 6. Memory Write Pipeline  (summarizer.py)       │
│    1) 최근 N턴 대화 요약                         │
│    2) 중요도 scoring (0.0~1.0)                  │
│    3) bge-m3 임베딩 생성                         │
│    4) ChromaDB 저장 (메타데이터 포함)            │
└─────────────────────────────────────────────────┘
```

## 컴포넌트별 설계

### 1. 페르소나 (캐릭터 YAML)

`CH_Haru.yaml` / `CH_Seonjae.yaml` 기준 현재 스키마입니다. (`character_schema.yaml` 계약 기준, 2026-04-02 재작성)

```yaml
id: Haru
name: 하루

description: >
  조용하고 차분한 태도로 대화를 이어가는 인물. 상대의 말에 섬세하게 반응한다.

# ── 말투 슬롯 (speech_style/* 학습 카테고리 대응) ────────────────
# prompt_build.py → _STYLE_PRESETS / _PERSONA_PRESETS로 해석
speech:
  formality: 반말
  style: blunt       # preset: 말이 짧고 직접적이다.
  persona: quiet_sensitive  # preset: 말수가 적지만 상대의 감정에 민감하게 반응한다.

# ── 성격 슬롯 (personality/* 학습 카테고리 대응) ──────────────────
# prompt_build.py → _PERSONALITY_PRESETS로 해석
personality: calm   # preset: 차분하고 안정된 태도. 쉽게 흔들리지 않는다.

# ── 친밀도 tier별 행동 슬롯 (affection/* 학습 카테고리 대응) ──────
# 학습 데이터 CATEGORY_ATTR과 텍스트 정확히 일치
affection:
  stranger:     "처음 만난 사이. 대화를 짧게 끊으려 하고 개인적인 반응을 거의 하지 않는다."
  acquaintance: "기본 대화는 가능하지만 경계가 있다. 개인적인 이야기는 아직 조심스럽다."
  familiar:     "조금 편해진 상태. 가끔 관심이 묻어나오지만 여전히 담담하다."
  friendly:     "자연스럽게 대화한다. 배려가 짧은 말 속에 드러나기 시작한다."
  close:        "배려가 자연스럽게 드러난다. 솔직한 반응을 자주 보인다."
  intimate:     "깊은 신뢰 상태. 감정을 짧게라도 솔직하게 표현한다."

# ── 감정 상태별 행동 슬롯 (emotion/* 학습 카테고리 대응) ──────────
emotion:
  happy:        "현재 기분이 좋은 상태. 반응이 약간 빨라지고 말이 조금 더 나온다."
  sad:          "현재 기분이 가라앉은 상태. 말이 짧아지고 주제를 돌리려 한다."
  angry:        "현재 화가 난 상태. 말이 차갑고 날카로워진다."
  annoyed:      "짜증난 상태. 반응이 건조하고 반문이 많아진다."
  curious:      "궁금증이 생긴 상태. 질문이 늘어나고 반응이 빨라진다."
  embarrassed:  "당혹스럽거나 부끄러운 상태. 말을 돌리거나 주제를 전환하려 한다."
  touched:      "마음이 움직인 상태. 짧은 침묵 후 말이 나온다."
  affectionate: "따뜻한 감정이 올라온 상태. 거리를 좁히려는 표현이 나온다."

# ── 규칙 ────────────────────────────────────────────────────────────
rules:
  - 캐릭터를 벗어나는 발언을 하지 않는다.
  - AI임을 언급하지 않는다.
  - 한국어만 사용한다. 영어 단어를 문장 중간에 섞지 않는다.
  - 문법에 맞지 않는 조사 결합을 사용하지 않는다.
  - '"물론이죠", "좋은 질문이에요", "도움이 되셨으면" 같은 표현을 사용하지 않는다.'

# ── 상태 초기값 및 전환 임계값 ────────────────────────────────────
state:
  mood_default: neutral
  affection_default: 30
  affection_thresholds:
    stranger:     [0,  15]
    acquaintance: [16, 30]
    familiar:     [31, 50]
    friendly:     [51, 70]
    close:        [71, 85]
    intimate:     [86, 100]
  mood_triggers:
    affectionate: ["좋아해", "보고 싶어", "곁에 있어줘"]
    touched:      ["고마워", "감동", "울컥", "따뜻하다"]
    happy:        ["좋아", "재밌어", "기뻐", "행복해"]
    curious:      ["궁금해", "어떻게 생각해", "알려줘"]
    sad:          ["슬퍼", "힘들어", "외로워", "우울해"]
    embarrassed:  ["당황", "어색해", "부끄"]
    annoyed:      ["짜증", "싫어", "그만해"]
    angry:        ["화났어", "열받아", "미치겠다"]
  affection_delta:
    affectionate: +5
    touched:      +4
    happy:        +3
    curious:      +1
    neutral:       0
    sad:           0
    embarrassed:  -1
    annoyed:      -3
    angry:        -6

# ── 대화 수위 파라미터 (_conv_hints()에서 tier별 자연어 지시문으로 변환) ──
conversation:
  response_length:
    stranger: 0.1  acquaintance: 0.2  familiar: 0.4  friendly: 0.55  close: 0.65  intimate: 0.75
  openness:
    stranger: 0.05  acquaintance: 0.15  familiar: 0.3  friendly: 0.45  close: 0.6  intimate: 0.75
  directness: 0.6

# ── 기억 표현 방식 ────────────────────────────────────────────────
memory_voice: "기억을 떠올릴 때 무심한 듯 언급한다. 예: '...그런 말 했었나.'"
```

### 2. 단기 메모리 vs 장기 메모리

| | 단기 (short_term.py) | 장기 (long_term.py) |
|---|---|---|
| 내용 | 최근 3~5턴 대화 원문 | 요약된 과거 에피소드 |
| 접근 방식 | 검색 없이 직접 삽입 | 시맨틱 유사도 검색 |
| 위치 | session.dialogue_log | ChromaDB |
| 용도 | "방금 한 말" 맥락 | "저번에 한 말" 맥락 |
| 트리거 | 매 턴 자동 | N턴 도달 시 요약→저장 |

**분리가 필요한 이유:**
- 단기 버퍼를 VDB로 검색하면 레이턴시 + 놓칠 가능성 발생
- 모두 컨텍스트에 포함하면 대화 누적 시 컨텍스트 오버플로우

### 3. 장기 메모리 VDB 저장 스키마

`M_schema.json` 확장 버전입니다.

```json
{
  "id": "mem_haru_012",
  "content": "사용자가 자신의 이름이 민준이라고 말했다. 바다를 좋아한다고 했다.",
  "metadata": {
    "character_id": "Haru",
    "turn_range": "3-8",
    "importance": 0.85,
    "tags": ["이름", "취미", "바다"],
    "location": "beach",
    "timestamp": "2025-01-18T09:30:00",
    "session_id": "session_001"
  }
}
```

**중요도(importance) 점수 기준:**

| 높음 (0.8+) → 저장 | 중간 (0.5~0.8) → 저장 | 낮음 (0.5 미만) → 저장 안 함 |
|---|---|---|
| 사용자 이름 / 고유 정보 | 감정적 사건 | 일상적 잡담 |
| 약속 / 선언 | 취향 / 선호도 | 단순 응답 ("응", "그래") |
| 갈등 또는 화해 | 반복된 화제 | 날씨 / 중립 소재 |

### 4. 시맨틱 검색 (RAG 및 장기 메모리)

**키워드 트리거 방식을 사용하지 않는 이유:**

```
저장된 기억: "사용자 이름 = 민준, 바다를 좋아함"

[키워드 방식의 문제]
입력: "내 이름 기억해?"   → "이름" 키워드 → 검색 미발동 가능
입력: "저번에 말했잖아"   → 키워드 없음  → 검색 미발동
입력: "민준"             → 키워드 매칭  → 검색 발동

[시맨틱 방식]
입력: "내 이름 기억해?"   → 임베딩 유사도 0.82 → 검색 발동 ✅
입력: "저번에 말했잖아"   → 임베딩 유사도 0.71 → 검색 발동 ✅
```

**검색 전략:**
- 매 턴 입력 텍스트를 bge-m3로 임베딩
- ChromaDB에서 top-2 결과 반환
- 유사도 임계값 0.52 미만이면 결과 없음 처리 (Layer C 생략) — `config.py vdb_threshold` 참조
- 결과 있을 경우 `memory_voice` 포맷으로 재서술 후 Layer C에 삽입

### 5. 캐릭터 관점 재서술

VDB에서 꺼낸 내용을 그대로 프롬프트에 삽입하면 모델이 데이터베이스 조회 결과처럼 처리합니다.

```
# 나쁜 예 — 데이터 문체
[기억] turn_3: 사용자가 "바다를 좋아한다"고 발언함. importance=0.85

# 좋은 예 — 캐릭터 내면 문체 (memory_voice 기반)
[어렴풋한 기억] 바다를 좋아한다고 했었지.
```

`memory_voice` 필드를 템플릿 변수로 활용해서 `prompt_build.py`에서 자동 변환합니다.

### 6. 토큰 예산

3B 모델 추론 컨텍스트 기준 (~4K)으로 설계합니다.

```
Layer A  캐릭터 시스템 프롬프트            300 tokens  [동적 — tier/mood/conv 반영]
Layer B  세계관 + Act + RAG 결과 병합      200~350 tokens  [동적]
Layer C  장기 메모리 VDB 결과 (top-2)      150 tokens  [동적, 없으면 0]
Layer D  최근 3~5턴 히스토리               300 tokens  [동적]
Layer E  현재 입력                         100 tokens  [가변]
생성 여유분                                 512 tokens
──────────────────────────────────────────────────────
합계                                     ~1,562 tokens  → 4K 이내 여유 충분
```

**초과 시 처리 (`PromptBuilder._layer_d()`):**
1. Layer D 히스토리를 5턴으로 시작
2. 예산 초과 시 3턴으로 축소
3. 그래도 초과 시 최근 1턴(메시지 쌍 2개)만 유지

---

### 7. 상태 관리 (mood / affection)

`session.py` 필드와 `agent/state.py` 전환 규칙입니다.

```python
# session.py
class ConversationSession:
    mood: str       = "neutral"
    affection: int  = 30          # 0~100 (CH_*.yaml affection_default에서 초기화)
    turn_count: int = 0
    dialogue_log: list = []

# agent/state.py — 6단계 affection 티어 (CH_*.yaml affection_thresholds에서 읽음)
# stranger(0~15) / acquaintance(16~30) / familiar(31~50)
# friendly(51~70) / close(71~85) / intimate(86~100)

# 8가지 mood 타입 (CH_*.yaml mood_triggers 우선순위 순)
# affectionate > touched > happy > curious > sad > embarrassed > annoyed > angry

# affection 증감: CH_*.yaml affection_delta 우선, 없으면 state.py _AFF_DELTA_DEFAULT 사용
_AFF_DELTA_DEFAULT = {
    "affectionate": +5, "touched": +4, "happy": +3, "curious": +1,
    "neutral": 0, "sad": 0, "embarrassed": -1, "annoyed": -3, "angry": -5,
}
```

상태 값은 `prompt_build.py`에서 Layer A 시스템 프롬프트에 동적으로 반영됩니다.
`conversation` 파라미터(`response_length`, `openness`, `directness`)는 `_conv_hints()`에서 tier별 자연어 지시문으로 변환되어 Layer A 평문 단락에 포함된다.

---

### 8. Post-processing 레이어

`router.py`의 `handle_turn()` 이후 실행되는 단계입니다.

```
LLM 응답 수신
    │
    ├─ [mood 업데이트]
    │    사용자 입력에서 감정어 감지 → mood 갱신
    │    (CH_*.yaml mood_triggers 키워드 매칭)
    │
    ├─ [affection 업데이트]
    │    갱신된 mood → affection_delta 조회 → 증감
    │    범위 0~100 클램프
    │
    └─ [메모리 쓰기 트리거]
         turn_count % N == 0 이면 summarizer.py 호출
         → 최근 N턴 요약 → 중요도 scoring → ChromaDB 저장
```

## 이질감 방지 설계 요약

이질감("AI스러운 느낌")을 줄이기 위한 설계 결정들입니다.

| 문제 | 원인 | 해결 |
|---|---|---|
| "물론이죠!" 같은 표현 | 기본 모델 응답 패턴 | `rules` 필드 + LoRA 파인튜닝 |
| 반말/존댓말 혼용 | 시스템 프롬프트 미흡 | `speech.formality` 슬롯 명시 + LoRA |
| 기억 없는 것처럼 행동 | 키워드 트리거 방식 실패 | 시맨틱 유사도 검색으로 교체 |
| 모든 턴 같은 감정 | 상태 관리 없음 | mood / affection 도입 |
| 기억 꺼내는 방식 어색 | VDB 원문 그대로 삽입 | 캐릭터 관점 재서술 |
| AI임을 언급 | 모델 기본 행동 | `rules` + 파인튜닝으로 차단 |
| 학습-추론 프롬프트 형식 불일치 | 학습 데이터 시스템 프롬프트가 추론 시 조립 순서와 다름 | character_schema.yaml 슬롯 체계 + rewrite_system_prompts.py 동기화 (2026-04-02) |
| 캐릭터 이름 학습 데이터에 하드코딩 | 다른 캐릭터 추가 시 학습 데이터 재생성 필요 | `{char_name}` 플레이스홀더 방식 도입 — 추론 시 YAML 이름 주입 |
| 기능 모드 자연어 출력 품질 부족 | 3B 모델 한국어 요약·응답 편차 | 프롬프트 엔지니어링 우선 적용, 부족 시 LoRA 검토 |
| 나레이션 문체 일관성 부족 | 호출마다 스타일 편차 | few-shot 프롬프트로 1차 제어, 불충분 시 별도 어댑터 학습 |

## 기능 모드 LLM 사용 원칙

기능 모드에서 LLM을 쓰는 지점은 두 종류로 분류된다.

### 파라미터 추출 (JSON 출력)

`file_rename`, `folder_classify`, `image_convert`, `local_search`, `web_search` 등
모든 도구의 1단계. `mode="function"` (greedy decoding + `repetition_penalty=1.3`)으로
JSON 한 줄을 추출한다. 출력 요구 정밀도가 낮아 **추가 LoRA 학습 불필요**.

### 자연어 생성 (요약·응답·변환)

| 기능 | 현재 구현 | 품질 결정 요인 | 권고 |
|---|---|---|---|
| 기능 결과 응답 | Layer F — 작업 목록 시스템 프롬프트 주입 | 캐릭터 말투 유지는 대화 LoRA가 처리 | 현행 유지 |
| 나레이션 | `conversation/narrator.py` (비활성화 중) | 문체 일관성 — 호출마다 편차 가능 | few-shot 프롬프트 우선, 부족 시 LoRA |
| prompt_convert 변환 | 크롤링→LLM 3단 파이프라인 | 이미지 모델 특화 지식은 사전학습 의존 | 프롬프트 구조 개선 우선 |
| 웹 검색 결과 요약 | 미구현 (링크 목록 반환) | 한국어 요약 품질 | 구현 시 few-shot 프롬프트로 시작 |

**적용 우선순위**: 프롬프트 엔지니어링(few-shot) → 프롬프트 구조 개선 → LoRA 추가
LoRA가 유효한 영역은 **대화(캐릭터 말투)** 뿐임을 원칙으로 한다.
기능 모드 품질 개선에 LoRA를 투입하기 전, 반드시 프롬프트 단계에서 충분히 시도해야 한다.

> 관련 문서: [docs/BUG/BUG_03.md](BUG/BUG_03.md) — 배포 환경 어댑터 on/off 선택지 및 프롬프트 엔지니어링 위치

### 파일별 구현 핵심 사항

#### `conversation/core/llm_client.py`
- `llama_cpp.Llama` 인스턴스, `n_ctx=4096`
- 스트리밍: `stream=True`, 토큰 단위 콜백
- `create_chat_completion` 사용 (ChatML 포맷 지원)
- 환경 변수 또는 `config.py`에서 모델 경로 수신

#### `conversation/core/session.py`
```python
@dataclass
class ConversationSession:
    character_id: str
    world_id:     str | None = None
    scenario_id:  str | None = None
    act_id:       str | None = None
    location:     str = ""                   # 현재 장소명 (YAML act.location 값)
    location_context: str | None = None      # 동적 장소 묘사 (_handle_location이 설정)
    mood:         str = "neutral"            # CH_*.yaml state.mood_default에서 초기화
    affection:    int = 30                   # CH_*.yaml state.affection_default에서 초기화
    turn_count:   int = 0
    dialogue_log: list = field(default_factory=list)  # 단기 버퍼용
    session_id:   str | None = None          # SessionManager가 부여하는 영속 ID
```

#### `conversation/core/prompt_build.py`
- Layer별 메서드 분리: `_layer_a()` ~ `_layer_d()` + `assemble()`
- 토큰 카운트: `llm.count_tokens` 주입, 미제공 시 한국어 간이 추정(~2자/토큰)
- 예산 초과 시 Layer D 턴 수 점진적 축소: `5턴 → 3턴 → 1턴(메시지 쌍 2개)`
- **Layer C 재서술**: `CH_*.yaml`의 `memory_voice` 필드를 템플릿으로 VDB 결과 감쌈
  ```
  # 변환 예시
  VDB raw : "사용자가 바다를 좋아한다고 말했다."
  Layer C  : "[어렴풋한 기억] 바다를 좋아한다고 했었지."
  ```
- **Layer A 상태 반영**: `session.affection` 값에 따라 응답 톤 지시문 분기 삽입

#### `memory/short_term.py`
- `session.dialogue_log` 전체를 반환 (n=None 기본값)
- 턴 수 축소는 호출자인 `PromptBuilder._layer_d()`에서 토큰 예산 기준으로 처리
- n을 명시하면 최대 n턴으로 자름 (현재 router는 n=None으로 호출)
- 반환 형식: `[{"role": "user", "content": ...}, {"role": "assistant", "content": ...}, ...]`

#### `memory/long_term.py`
- ChromaDB collection명: `{character_id}_memory`
- 임베딩 모델: `SentenceTransformer("BAAI/bge-m3")`
- 저장: `M_schema.json`의 `properties` 구조 준수
- 검색:
  ```python
  results = collection.query(
      query_embeddings=[query_vec],
      n_results=2,
      where={"importance": {"$gte": 0.5}}
  )
  # cosine similarity < 0.7이면 빈 리스트 반환
  ```

#### `memory/summarizer.py`
- 트리거: `session.turn_count % N == 0` (N = config, 기본 10)
- 요약: LLM에 별도 짧은 프롬프트 요청 (`max_tokens=150`)
- 중요도 scoring: `M_schema.json`의 `importance_rules` 기준으로 규칙 기반 판정
- 0.5 미만이면 ChromaDB 저장 생략

#### `conversation/core/router.py` — `handle_turn()` 흐름
```
0. _handle_location(user_input)      # 장소 이동 감지 → session.location_context 갱신
1. short_buf  = short_term.get(session)              # dialogue_log 전체
2. vdb_result = long_term.query(user_input, char_id) # 장기 메모리 (Layer C)
3. rag_result = rag.query(user_input)                # 세계관 RAG (Layer B 병합)
4. messages   = prompt_build.assemble(short_buf, vdb_result, rag_result)
5. response   = llm.generate(messages)
6. state_mod.update_mood(session, user_input)        # 사용자 입력 기준 mood 갱신
   state_mod.update_affection(session, mood)
7. session.add_turn(user_input, response)
8. summarizer.check_trigger(session)                 # N턴 조건 충족 시 VDB 저장
return response
```

#### `agent/state.py`
- `CH_*.yaml`의 `state.mood_triggers` 로드
- `affection` 변화: `CH_*.yaml affection_delta` 우선, 없으면 `_AFF_DELTA_DEFAULT` 폴백, 범위 0~100 클램프
- `mood` 변화: 사용자 입력에서 `mood_triggers` 키워드 매칭 (어근 형태 — 어미 변화 포괄)

## 로그 데이터 학습 활용 계획 (DPO — 보류 중)

> 현재 로그 데이터 수가 적어 보류. 데이터 누적 후 적용 예정.

### 방향: SFT 선행 → DPO 후속

| 단계 | 방법 | 데이터 소스 | 상태 |
|---|---|---|---|
| 1 | **SFT** (현재 방식) | `training/data/**/*.jsonl` | 진행 중 |
| 2 | **SFT 보강** | `training/log/feedback_pos/` + `training/log/daily/` → `training/data/log/` 변환 | 로그 충분 시 |
| 3 | **DPO** | `feedback_pos` (chosen) + `feedback_neg` (rejected) 쌍 구성 | 미구현 |

### DPO 데이터 구조 (미래 참고)

```json
{
  "prompt": "나 오늘 진짜 힘들었어.",
  "chosen": "뭐가 있었어.",
  "rejected": "그렇군요, 힘드셨겠네요. 무슨 일이 있었나요?"
}
```

- `feedback_pos/` → chosen (사용자가 좋다고 판단한 응답)
- `feedback_neg/` → rejected (수정 요청된 원래 응답)
- 쌍이 되는 로그가 충분히 쌓이면 `training/dpo/` 디렉토리 신설 후 별도 파이프라인 구성
