    │
    ├─ ④ [RAG] WorldRetriever.query()

    │

## 1. Conversation

### 현재 상태
- `router.py::handle_turn()`이 전체 파이프라인을 순서대로 실행. 완전 동작 중.
- Layer 구조 (PromptBuilder): A(캐릭터 시스템) 300토큰 + B(세계관+act) 200토큰 + C(VDB) 150토큰 + D(단기히스토리) 300토큰 + 생성 512토큰.
- mood 8종 / affection 6단계 / location_context 동적 업데이트 모두 연결됨.
- narrator.py 비활성화 — 장소 이동 시 장면 묘사 없이 바로 대화 진행.
- **`PromptBuilder._layer_a()` 개선** (2026-03-29):
  - YAML `rules` 필드를 직접 join하여 시스템 프롬프트에 반영 (이전: 하드코딩 요약으로 무시됨)
  - 이름 형식 `"너의 이름은 X이다."` → `"너는 X이다."` 로 학습 데이터 포맷과 통일
  - CH_Haru.yaml / CH_Seonjae.yaml 에 추가된 언어/문법/stranger tier 규칙이 Layer A에 포함됨

### 맞물림 이슈
- **mood 트리거가 사용자 입력 기준**: LLM 응답이 아닌 사용자 발화 키워드로 mood를 판단. 캐릭터가 실제로 어떤 감정을 표현했는지는 반영 안 됨.
- ~~**session에 location 필드 없음**~~: ✅ `session.location` 필드 추가, router.py에서 장소 이동 시 자동 갱신, bridge.py YAML 역참조 제거 (2026-03-22).
- **UI와의 연결**: `bridge.py::_sync_state()`가 응답 후 mood/배경 변화를 감지해 시그널 emit — 정상 연결.

### 개선 가능성
- narrator.py 재활성화: 장소 이동 시 3~5문장 묘사를 삽입하면 몰입감 향상. 대화 품질 안정화 후 router.py 98줄 주석 해제로 즉시 적용 가능.
- ~~session에 `location` 필드 추가~~: ✅ 완료 (2026-03-22).
- mood를 LLM 응답 분석 기반으로 전환하면 정확도 향상 (현재는 단순 키워드 매칭).

## 2. Memory / 학습 로그 → VDB 저장

### 현재 상태

**대화 메모리 (런타임)**
- **단기(short_term)**: `session.dialogue_log` 슬라이딩 윈도우 (최근 N턴). 간단하고 완전 동작.
- **장기(long_term)**: ChromaDB VDB. 요약 후 저장, 관련 입력 시 검색해 Layer C에 삽입. 완전 동작.
- **요약(summarizer)**: N턴마다 LLM으로 대화 요약 → 키워드 중요도 scoring → VDB 저장. 완전 동작.
  - **이름 importance 보완** (2026-03-28/29): `_HIGH_KEYWORDS`에 "이름:", "이름은", "이름이" 추가. 시스템 프롬프트에 "이름: [이름]" 형태 포함 지시 추가. mid 키워드 분기 dead code 수정(`score < 0.5` → `score == 0.5`).
  - 중요도 기준: high 키워드(이름/약속/갈등) → 0.85 / mid 키워드(취미/감정/날짜) → 0.6 / 무키워드 → 0.5

**학습 로그 수집 (ConversationLogger)**
- 대화 중 유저 발화 키워드를 분석해 카테고리(daily/emotion/memory/advice/persona/negative/positive) 분류.
- 트리거 조건: 키워드 매칭 즉시 flush / 반복 루프(assistant 응답 3회 동일) 즉시 flush / CHUNK_SIZE 턴 도달 / 세션 종료.
- Jaccard 유사도 기반 중복 제거 후 `training/log/{category}/YYYY-MM-DD.jsonl` 저장.
- `scripts/build_dataset.py`가 이 로그를 읽어 ChatML 포맷 학습 데이터로 변환 → LoRA 학습에 사용.

### 맞물림 이슈
- 장기 메모리 컬렉션명이 `{character_id.lower()}_memory` — 캐릭터 변경 시 새 컬렉션으로 자동 전환됨. 정상.
- 요약 트리거 간격(`memory_trigger_n=10`)이 고정값 — 대화 밀도에 따라 조절 여지 있음.
- **단기↔장기 경계 불명확**: 단기 버퍼(Layer D)와 장기 VDB 검색 결과(Layer C)가 중복 내용을 담을 수 있음.
- ~~**`build_dataset.py`의 glob 버그**~~: ✅ `glob("**/*.jsonl")` 수정 완료 (2026-03-22).
- `training/log/advice/`, `persona/` 폴더 미존재 → 해당 카테고리 로그 수집 안 됨.

### 개선 가능성
- 중요도 threshold(현재 0.5)를 캐릭터별로 조정 가능하게 하면 캐릭터마다 기억 민감도 차별화 가능.
- 요약 시 affection·mood 변화 이벤트를 별도 메타데이터로 저장하면 "우리가 처음 친해졌을 때" 같은 맥락 검색 품질 향상 가능.
- 로그 카테고리에 mood별 감정 반응 데이터 추가(affectionate/touched/curious/embarrassed/angry) 시 감정 표현력 향상.

### 테스트 구현 항목 (`tests/`)
- `test_conversation_logger.py`: 카테고리 분류 키워드 매칭, Jaccard 중복 제거, flush 트리거 조건
- `test_summarizer.py`: N턴 트리거, 중요도 scoring 임계값, VDB 저장 형식
- `test_build_dataset.py`: glob 수정 후 서브폴더 파일 탐색 확인, ChatML 변환 형식 검증

## 3. VDB

### 현재 상태
- ChromaDB PersistentClient 사용. `chroma_dev/` 로컬 디스크 저장.
- 두 가지 용도로 분리:
  - `world_knowledge` 컬렉션 → 세계관 RAG
  - `{char_id}_memory` 컬렉션 → 캐릭터별 장기 메모리
- 임베딩 모델: BAAI/bge-m3 (384차원, 한국어 최적화). cosine 유사도. threshold 0.52.

### 맞물림 이슈
- **동일 ChromaDB 인스턴스 공유**: 세계관 RAG와 장기 메모리가 같은 DB를 쓰지만 컬렉션이 달라 분리는 됨. 단, DB 파일 하나가 깨지면 전체 영향.
- **배포 환경은 `chroma_deploy/` 별도 경로**: dev↔deploy 데이터 분리는 되어 있음.
- `ui_test` 환경에서는 임베딩 모델 None → VDB 기능 동작 안 함. 테스트 중 장기 메모리/RAG 검색 불가.

### 개선 가능성
- 세계관 RAG용 컬렉션과 메모리 컬렉션을 물리적으로 다른 DB로 분리하면 백업/마이그레이션 용이.
- threshold를 쿼리 유형(세계관 vs 메모리)별로 다르게 설정할 여지 있음 (현재 둘 다 0.52).

### 테스트 구현 항목 (`tests/`)
- `test_vdb.py`: 저장→검색 왕복, threshold 필터 동작, 컬렉션 분리(world_knowledge vs _memory)
- ui_test 환경에서 임베딩 None일 때 조용한 폴백(검색 결과 빈 리스트 반환) 확인

## 4. RAG

### 현재 상태
- `rag/index.py`: `sources/world/*.md` 읽기 → 400자 고정 크기 청킹(overlap 50) → bge-m3 임베딩 → ChromaDB 저장.
- `rag/retrieve.py::WorldRetriever.query()`: 매 턴 자동 실행. 상위 2개 검색 후 threshold 필터.
- `rag/world_nav.py`: 이동 의도 감지 → YAML acts 매칭 or LLM으로 새 장소 생성 → VDB에 동적 저장.

### 맞물림 이슈
- **세계관 문서가 현재 1개**(seaside_world): 청킹 불일치 문제가 실질적 영향을 주지 않음. 문서가 늘면 semantic chunking 필요.
- **world_nav가 LLM을 2회 호출**: detect_move_intent(LLM) + find_or_create_location(LLM). 매 턴 실행 시 지연 가능성.
- 동적 생성된 장소가 VDB에 저장되지만 YAML에는 반영 안 됨 → 세션 재시작 시 YAML에 없는 장소가 VDB에만 남음.

### 개선 가능성
- 고정 크기 → `##` 헤더 기준 semantic chunking으로 전환 시 의미 단위 보존 향상 (현재 보류).
- 이동 의도 감지를 LLM 호출 없이 키워드 사전 기반으로만 처리하는 fast-path를 두면 레이턴시 감소.
- 동적 생성 장소를 YAML에도 병합 저장하는 로직 추가 시 세션 간 일관성 유지 가능.

### 테스트 구현 항목 (`tests/`)
- `test_rag_index.py`: 청킹 결과 청크 수·길이 확인, ChromaDB 저장 확인
- `test_rag_retrieve.py`: threshold 이상/이하 문서 필터링, 상위 N개 반환 확인
- `test_world_nav.py`: 이동 의도 감지 키워드 매칭, YAML acts 매칭 로직

## 7. Tools

### 현재 상태
대화 이외의 보조 기능을 LLM 도구 호출 형식으로 제공. `agent/core.py`의 `handle_input(mode="function")`을 통해 진입하며, 슬래시 명령어(`mode="chat"`)와 별도 계층.

**트리거 방식**: 사용자 발화에서 자연어 키워드를 감지(`agent/core.py::_KEYWORDS`) → 도구 선택 → LLM이 파라미터 JSON 추출 → rule-based 실행.

| 도구 | 파일 | 트리거 키워드 | 기능 |
|---|---|---|---|
| `folder_classify` | `tools/folder/classifier.py` | `분류`, `정리`, `폴더` | 디렉토리 파일을 확장자/MIME 타입 기준으로 폴더 분류 |
| `image_convert` | `tools/folder/converter.py` | `이미지`, `image`, `png`, `jpg`, `jpeg`, `webp`, `bmp`, `tiff` | 이미지 포맷 변환 (Pillow: jpg/png/webp/bmp/tiff) |
| `file_rename` | `tools/folder/renamer.py` | `이름`, `rename`, `renamer`, `파일명` | 규칙 기반 파일 일괄 이름 변경 (7가지 규칙, glob 패턴) |
| `prompt_convert` | `tools/prompt_converter.py` | `프롬프트`, `prompt`, `명확하게`, `간결하게`, `상세하게`, `질문형`, `지시형` | 프롬프트 스타일 변환 (5종) |
| `local_search` | `tools/search/local_search.py` | `검색`, `search`, `찾아`, `파일 찾` | SQLite FTS5 기반 로컬 파일 전문 검색 (증분 인덱싱) |
| `web_search` | `tools/search/web_search.py` | `인터넷`, `웹 검색`, `web`, `구글`, `검색해줘`, `찾아봐` | ddgs API 기반 웹 검색 (구 duckduckgo-search → ddgs 패키지 이름 변경 반영) |

> 키워드 우선순위: `prompt_convert` → `file_rename` → `folder_classify` → `image_convert` → `web_search` → `local_search` 순으로 먼저 매칭된 도구가 선택됨. `_KEYWORDS` 리스트 순서 기준.
