# UI/UX 개선 3차 — 피드백 및 방향 검토

> 작성일: 2026-04-07
> 상태: 방향 검토 (구현 계획 미확정)
> 목적: 사용자가 제기한 7개 항목의 현황 진단 + 개선 방향 제안

---

## 1. 세션 구분

### 현황 진단

- `SessionManager`가 `data/sessions/{char_id}/` 에 세션별 JSON을 저장하는 구조는 이미 존재한다.
- `bridge.listSessions(char_id)`로 세션 목록 조회가 가능하고, `bridge.newSession(keep_memory)`로 새 세션을 생성한다.
- 그러나 SettingsPanel에 세션 목록을 표시하는 UI가 없다. "새 대화 시작" 버튼 2개만 존재하며, 세션 간 전환(스위치)이 아닌 무조건 새 세션 생성만 된다.
- 세션 생성 기준(캐릭터 변경 / 세계관 변경)도 현재는 코드 수준에서 명확히 구분되지 않는다.

### 문제점

1. **목록이 안 보인다** — 현재 어떤 세션이 있는지 UI에서 확인 불가.
2. **전환이 안 된다** — 기존 세션으로 돌아가는 경로가 없고 새 세션만 생성된다.
3. **세션 생성 기준이 불명확** — 캐릭터/세계관 변경이 항상 새 세션을 만드는지, 아니면 기존 세션으로 전환하는지 구분이 안 된다.
4. **로그 표시 정책** — 세션 전환 시 최근 10턴만 보이는 규칙이 코드에 없다.

### 제안 방향

- SettingsPanel 세션 관리 섹션을 **세션 목록 + 선택 전환** 형태로 개편한다.
  - 세션 카드: 캐릭터 이름 / 세계관 / 마지막 대화 날짜 / 턴 수 표시
  - 최대 3개 유지 (3개 초과 시 가장 오래된 세션 자동 삭제 또는 사용자 선택 삭제)
- 세션 전환 시 `bridge`가 해당 세션의 최근 10턴만 `chatModel`에 로드한다. (이전 로그는 VDB에 이미 요약·저장된 상태)
- **세션 생성 기준 명확화:**
  - 동일 캐릭터 + 동일 세계관 조합의 세션이 이미 존재하면 → 전환만
  - 처음 조합이면 → 새 세션 생성
  - 세션은 `(char_id, world_id)` 쌍으로 식별
- `bridge.switchSession(session_id)` 슬롯 신규 추가 필요.

## 2. 시나리오 항목 → 세계관 항목

### 현황 진단

- SettingsPanel의 섹션 이름이 "시나리오"로 되어 있다.
- 현재 구조는 세계관 → 시나리오 → act 3단계 계층인데, act별로 항목이 나열되어 있어 어떤 세계관에 속하는지 파악이 어렵다.
- `W_sea.yaml` 하나만 존재하며, 복수 세계관 선택 경험이 없어 UI 미숙.
- `world_knowledge` ChromaDB 컬렉션이 세계관의 영구지식 저장소 역할을 하는데, `rag/sources/world/*.md`가 인덱싱 소스다.

### 문제점

1. **이름 혼동** — "시나리오"는 내부 용어(scenario_id)이고, 사용자에게 노출되는 용어는 "세계관"이어야 한다.
2. **계층 UX 부재** — 세계관 선택 → 장소(act) 선택의 2단계 드릴다운이 없고 평면 목록이다.
3. **기본 세계관 미지정** — 앱 구동 시 `W_sea.yaml`이 default로 자동 선택되어야 하는데, 코드상 명시적 default 처리가 없다.
4. **세계관 = 세션 경계** — 같은 세계관 내 장소 이동은 세션 변경이 아님을 UI가 표현하지 못한다.

### 제안 방향

- SettingsPanel 섹션명: "시나리오" → **"세계관"**
- 2단계 구조로 개편:
  1. **세계관 목록** — 클릭 시 해당 세계관 내 장소 목록 펼침 (아코디언)
  2. **장소(act) 선택** — 선택 시 `bridge.changeWorld()` 호출, 동일 세계관이면 세션 유지
- 앱 구동 시 `W_sea.yaml`의 `world_id`를 default로 자동 지정. `config.py`에 `default_world_id = "seaside_world"` 추가 검토.
- 장소 항목에 "(현재 위치)" 배지 표시.
- 세계관 추가는 DB 섹션에서 처리 (아래 3번 참조).

## 3. 데이터 항목 → DB 항목

### 현황 진단

- 섹션명이 "데이터"로 되어 있고, 관리자 패널 진입점도 함께 있다.
- `MemoryDBPanel`이 ChromaDB의 `{char_id}_memory` 컬렉션을 카드 목록으로 표시하는데, 어떤 컬렉션인지, 어떤 세션·캐릭터에서 생성됐는지 컨텍스트가 없다.
- 세계관 지식(`world_knowledge`)과 프롬프트 가이드(`prompt_guides`) 컬렉션은 현재 DB 패널에서 아예 보이지 않는다.

### 문제점

1. **섹션명** — "데이터" → **"DB"**로 변경 필요.
2. **관리자 패널** — 대화창 상단에서 접근 가능하므로 이 섹션에서 제거.
3. **컬렉션 가시성 부재** — Memory / World / Prompt 3종 컬렉션 중 Memory만 보인다.
4. **메타 정보 부재** — 각 항목이 어떤 세션, 어떤 캐릭터, 몇 턴째 생성인지 표시 안 됨.
5. **세계관 지식 편집 불가** — `rag/sources/world/*.md` 원문을 UI에서 열람·편집하는 경로가 없다.
6. **세계관 추가 경로 없음** — 사용자가 새 세계관을 대화로 만들어 저장하는 흐름이 미구현.
7. **프롬프트 컬렉션** — `prompt_guides` ChromaDB 컬렉션이 있으나 UI에서 접근 불가.

### 제안 방향

**섹션 이름:** "데이터" → **"DB"**, 관리자 패널 항목 제거

**DB 패널 계층 구조 (시각적):**
```
DB
├── Memory
│   ├── short  (현 세션 단기 버퍼 — 읽기 전용 미리보기)
│   └── long   (VDB 장기 기억 — 조회·편집·삭제)
├── World (세계관 지식)
│   ├── seaside_world  → rag/sources/world/ 원문 열람·편집
│   └── + 새 세계관 추가
└── Prompt (프롬프트 가이드)
    └── {model_name}  → 저장된 가이드 열람·편집·삭제
```

**세계관 지식 편집:**
- `World` > 세계관 선택 시 해당 `rag/sources/{world_id}/` 폴더의 `.md` 파일을 탭 또는 목록으로 열람.
- 인라인 텍스트 편집 → 저장 시 `.md` 파일 덮어쓰기 + ChromaDB 재인덱싱(`index_world()`) 자동 호출.

**새 세계관 추가:**
- "+ 새 세계관" 버튼 → 이름 입력 창.
- 생성 시 `rag/sources/{world_id}/` 폴더 + 빈 `story.md / place.md / culture.md` 생성.
- 이후 대화를 통해 내용을 채우면, 대화 내용을 요약해 해당 `.md`에 추가하는 흐름 (중기).

**Memory 메타 정보 개선:**
- 각 기억 카드에 `char_id` / `session_id` / `저장 일시` / `중요도` 표시.
- 현재 `importance`, `created_at`, `keywords` 메타데이터가 이미 ChromaDB에 저장되어 있으므로 UI에 노출만 하면 된다.

## 4. 관리 모드

### 현황 진단

- 친밀도 슬라이더(`affSlider`)가 `DragHandler` 없이 `Slider` 컨트롤로 구현되어 있다.
- AdminPanel이 `anchors.fill: parent`인 채로 Window 위에 올라가 있어, Slider 드래그 이벤트가 Window의 `DragHandler`와 충돌한다. 결과적으로 슬라이더를 드래그하면 창 자체가 움직인다.
- 응답 길이(`response_length`) / 감정 개방도(`openness`)가 Slider 형태로 구현되어 있다.
- 직접성(`directness`)은 단일 Slider.
- 헤더에 "(드래그로 이동)" 문구가 추가되어 있다 (이전 수정에서 추가됨).

### 문제점

1. **Slider ↔ Window DragHandler 충돌** — 핵심 버그. 슬라이더 상호작용이 불가능.
2. **응답 길이·감정 개방도** — 슬라이더보다 선택지가 직관적 (짧게/보통/길게 / 낮음/보통/높음).
3. **"(드래그로 이동)" 문구** — 불필요, 제거 필요.

### 제안 방향

**Slider 충돌 해결:**
- AdminPanel 모달 내부에서 `DragHandler`가 아닌 `MouseArea`의 `onPositionChanged`로 이동을 처리한다 (이미 이렇게 구현됨).
- 문제의 핵심은 Window 레벨 `DragHandler`가 AdminPanel의 `MouseArea` 이벤트보다 먼저 캡처하는 것.
- 해결책: AdminPanel이 보일 때 Window의 `DragHandler.enabled = false`로 전환. 닫히면 재활성.

**응답 길이 / 감정 개방도:**
- 현재: Slider (0.0~1.0 연속값, 티어별)
- 변경: 3단계 버튼 선택 — `[짧게] [보통] [길게]` / `[낮음] [보통] [높음]`
- 내부 저장값은 `low=0.3 / mid=0.6 / high=0.9` 등 고정 매핑.
- `setConvParam()` 슬롯의 인터페이스는 유지하되 값만 고정 3단계로 제한.

**직접성:** 현행 Slider 유지.

**헤더 문구:** "관리자  (드래그로 이동)" → "관리자" 로 복원.

## 5. 캐릭터 항목

### 현황 진단

- CH_Haru.yaml, CH_Seonjae.yaml, CH_default.yaml 3개가 존재.
- `CharacterSelectPanel`에서 모든 `CH_*.yaml`이 목록에 표시된다.
- `CharacterCreatePanel.qml`이 신규 추가되었으나 실제 저장·영속화 로직이 완결됐는지 불확실.
- 캐릭터 YAML이 `conversation/character/` 폴더에 저장되며, 앱 재시작 시 폴더를 스캔하므로 파일만 남아있으면 영속된다.

### 문제점

1. **불필요한 캐릭터** — CH_Seonjae가 목록에 노출되나 사용 목적이 없음.
2. **캐릭터 생성 흐름** — CharacterCreatePanel에서 생성한 YAML이 실제로 저장되고, 재시작 후에도 목록에 나타나는지 검증 필요.
3. **캐릭터 삭제** — UI에서 캐릭터를 삭제하는 경로가 없다.

### 제안 방향

- **CH_Seonjae.yaml 제거** 또는 `_excluded/`로 이동해 기본 목록에서 숨김.
- **캐릭터 생성 흐름 검증:**
  - CharacterCreatePanel 저장 → `conversation/character/CH_{id}.yaml` 실제 기록 확인.
  - 앱 재시작 후 목록 재스캔으로 영속 확인.
- **캐릭터 삭제:**
  - CharacterSelectPanel 카드에 "삭제" 버튼 추가.
  - 현재 활성 캐릭터는 삭제 불가 처리.
  - YAML 파일 삭제 + 해당 캐릭터의 VDB 컬렉션 + 세션 파일 함께 정리.
- **캐릭터 생성 메뉴 진입점:**
  - 현재 CharacterSelectPanel의 "+" 버튼이 이미 존재 → 흐름은 유지.
  - 생성 완료 후 CharacterSelectPanel 목록 자동 갱신 확인.

## 6. 모드 전환 및 절전

### 현황 진단

- 대화 모드 / 기능 모드 분리는 `bridge.sendMessage(text, mode)` 레벨에서 처리.
- 기능 모드 중 폴더 분류 / 파일 검색 / 파일 변환은 LLM 없이 동작 (태그 클릭 → bridge 슬롯 직접 호출).
- 단, `prompt_convert` 도구는 여전히 LLM 호출 경로를 거친다.
- 절전 모드 (LLM 언로드 → 재로드)는 현재 없다. `_unload_llm()`은 캐릭터 전환 시에만 호출.

### 문제점

1. **불필요한 LLM 호출** — 기능 모드 중 rule-based 동작(폴더 정리 등)에서 LLM이 호출되지 않도록 이미 분리되어 있으나, 코드 경로 검증 필요.
2. **절전 모드 없음** — PIP 대기 중에도 모델이 VRAM에 상주. 3B int4 ≈ 2GB 점유 지속.
3. **재로드 비용** — `_load_transformers()` 초기 로딩이 약 10~15초 소요. 절전 후 재가동 시 사용자가 대기해야 한다.

### 제안 방향

**LLM 최소 호출 정책:**

| 상황 | 현재 | 방향 |
|---|---|---|
| 폴더 분류 / 파일 변환 / 로컬 검색 | LLM 미호출 (이미 분리) | 유지 |
| prompt_convert | LLM 호출 | 유지 (파라미터 추출 필요) |
| 대화 모드 | LLM 호출 | 유지 |
| PIP 대기 | LLM 상주 | 선택적 해제 검토 |

**절전 모드 방향 (조건부 검토):**
- **장점:** PIP 대기 시 VRAM 2GB 반환, RAM 절감.
- **단점:** 재로드 10~15초 → 첫 메시지 응답 전 사용자가 기다려야 함.
- **절충안:** 즉시 언로드가 아닌, **일정 시간(예: 10분) 비활성 후** 언로드. 메시지 입력 감지 시 "모델 로딩 중..." 상태 표시 후 재로드.
- `QTimer`로 마지막 sendMessage 시각 추적 → 타임아웃 시 `_unload_llm()` 호출.
- 재활성 시 `LLMWorker.run()` 시작 전 로드가 필요하므로, `LLMWorker`에 "로딩 중" 상태 시그널 추가.
- **우선순위는 낮음** — RAM이 실제 문제가 되는 시점에 구현 검토.

## 7. 상태창 (CharacterStatusPanel) 크기 문제

### 현황 진단

- `CharacterStatusPanel`의 `width: 260`이 고정값.
- `height: contentLayout.implicitHeight + 32`로 내용 기반 높이이나, 내부 텍스트가 260px를 초과할 때 잘림 발생.
- 현재 상태창에 친밀도, tier, 감정, 대화 횟수 등이 포함. 텍스트가 길어지면 삐져나옴.

### 문제점

1. **고정 너비** — 260px가 일부 텍스트에 좁음.
2. **텍스트 클리핑** — `clip: true` 없이 텍스트가 영역 밖으로 나옴.
3. **스크롤 없음** — 내용이 많아져도 스크롤 불가.

### 제안 방향

- `width` 확대: 260 → **300** (현재 앱 너비 432에서 여유 있음).
- 내부 텍스트에 `elide: Text.ElideRight` + `wrapMode: Text.WordWrap` 적용.
- 높이가 일정 기준 초과 시 `ScrollView` 적용.
- 단기적으로는 너비 확대 + wrapMode 적용만으로 대부분 해결 가능.
