## 항목 1 — 메모리 DB 정리 및 구조 명확화

### 1-1. 현재 상태의 문제

**DB 오염**:
- 개발 초기부터 현재까지 ChromaDB를 한 번도 초기화하지 않음
- MVP 단계 대화, 구버전 어댑터(LoRA v3~v8) 적용 모델의 응답이 VDB에 그대로 존재
- `long_term.query()`는 품질/날짜 필터 없이 코사인 유사도만으로 반환 → 낮은 품질의 요약이 주입됨

**long_term / short_term 경계 불명확**:
- `short_term.get_recent()` = `session.dialogue_log` 슬라이싱 (in-context, 런타임 전용)
- `long_term` = ChromaDB VDB (영속, 세션 간 공유)
- 현재 코드에서 두 레이어의 역할은 명확하지만, 요약 저장 시점(`trigger_n=10`)이 짧아 short_term과 long_term의 내용이 중복될 수 있음

#### 1-2-C. 검색 시 날짜 필터 (선택적 적용)

```python
# long_term.query() — 너무 오래된 기억은 제외 옵션
def query(self, text: str, character_id: str, since_days: int | None = None) -> list[str]:
    where = {"character_id": character_id}
    if since_days:
        cutoff = (datetime.now(timezone.utc) - timedelta(days=since_days)).isoformat()
        where = {"$and": [{"character_id": character_id}, {"timestamp": {"$gte": cutoff}}]}
```

초기에는 `since_days=None` (전체 조회 유지), 오염 재발 시 `since_days=30` 등으로 조정.

## 항목 2 — mood 지속성 및 키워드 확장

### 2-1. 현재 구조의 문제

`update_mood()`는 매 턴 `new_mood = "neutral"` 로 초기화 후 키워드 매칭:

```python
new_mood = "neutral"                          # 매 턴 리셋
for mood_name, keywords in triggers.items():
    if any(kw in user_input for kw in keywords):
        new_mood = mood_name
        break
session.mood = new_mood                       # 키워드 없으면 항상 neutral
```

결과:
- 사용자가 "좋아" → `happy`
- 다음 턴에서 "어디 갈까" → 키워드 없음 → 즉시 `neutral`로 복귀
- 감정이 축적되거나 유지되지 않음 — 캐릭터가 항상 무감각하게 보임
- `affection_delta`가 mood에 의존하므로, mood가 항상 neutral = affection도 거의 안 변함

**MookHyeon 추가 문제**: 키워드 목록이 Haru보다 절반 수준 (7개 mood × 2키워드 = 14개). 자연스러운 대화에서 거의 매칭이 일어나지 않음.

### 2-2. 수정 방향

#### 2-2-A. `state.py` — mood 감쇠(decay) 도입

neutral이 아닌 mood가 발동된 이후, 키워드 없는 턴이 지속되면 점차 neutral로 복귀.
급격한 리셋 대신 자연스러운 감정 소멸.

```python
# session에 _mood_turns 카운터 추가 (현재 mood 유지 턴 수)
# update_mood() 수정안

_MOOD_DECAY_TURNS = 3   # 키워드 없는 턴이 N번 연속되면 neutral로 복귀

def update_mood(session: ConversationSession, user_input: str, character: dict) -> str:
    triggers = character.get("state", {}).get("mood_triggers", {})
    decay_turns = character.get("state", {}).get("mood_decay_turns", _MOOD_DECAY_TURNS)

    # 키워드 매칭
    matched_mood = None
    for mood_name, keywords in triggers.items():
        if any(kw in user_input for kw in keywords):
            matched_mood = mood_name
            break

    if matched_mood:
        # 새 mood 발동 → 카운터 초기화
        session.mood = matched_mood
        session._mood_hold = decay_turns
    elif session.mood != "neutral":
        # 유지 카운터 감소
        hold = max(0, getattr(session, "_mood_hold", 0) - 1)
        session._mood_hold = hold
        if hold == 0:
            session.mood = "neutral"
    # neutral이면 그대로 유지

    return session.mood
```

#### 2-2-B. 캐릭터 YAML — `mood_decay_turns` 필드 추가

```yaml
state:
  mood_decay_turns: 3   # Haru: 감정 유지 3턴 (무뚝뚝하지만 짧게 지속)
```

```yaml
state:
  mood_decay_turns: 4   # MookHyeon: 감정이 더 오래 남는 편
```

#### 2-2-C. `CH_MookHyeon.yaml` — 키워드 확장

현재 7개 mood에 2개씩 = 14개 키워드. 최소 각 mood당 5~7개로 확장.

```yaml
mood_triggers:
  affectionate: ["좋아해", "보고 싶어", "곁에 있어줘", "같이 있고 싶어", "의지가 돼"]
  touched:      ["고마워", "감동", "마음에 닿", "따뜻하다", "위로가 됐어"]
  happy:        ["좋아", "재밌어", "웃겨", "기뻐", "즐거워", "좋은데"]
  curious:      ["궁금해", "어떻게 생각해", "왜", "알려줘", "뭔가요"]
  sad:          ["슬퍼", "힘들어", "외로워", "우울해", "눈물", "지쳐"]
  embarrassed:  ["당황", "어색해", "뭐래", "부끄", "갑자기"]
  annoyed:      ["짜증", "싫어", "그만해", "됐어", "신경꺼", "왜 그래"]
  angry:        ["화났어", "열받아", "미치겠다", "진심이야", "말이 돼"]
```

#### 2-2-D. `_mood_hold` 세션 복원

`SessionState` ↔ `ConversationSession` 동기화 경로에서 `_mood_hold` 값을 복원해야 함.
단순히 `SessionState`에 `mood_hold: int = 0` 필드 추가 후 `restore_session_from_state()` 에서 복사.

### 2-3. 기대 효과

```
# 수정 전
user: "좋아" → mood: happy
user: "어디 가자" → mood: neutral   ← 즉시 리셋

# 수정 후 (decay_turns=3)
user: "좋아" → mood: happy, _mood_hold=3
user: "어디 가자" → mood: happy, _mood_hold=2
user: "그렇구나" → mood: happy, _mood_hold=1
user: "..." → mood: neutral           ← 자연스럽게 소멸
```

affection_delta도 3턴간 `happy(+1~+3)`가 누적 적용 → 더 의미 있는 aff 변화.

## 항목 3 — 요약 프롬프트 구조화

### 3-1. 현재 문제

```python
"다음 대화를 1~3문장으로 간결하게 요약해. "
"사용자에 대해 알게 된 정보(이름, 취미, 감정, 사건)를 중심으로 써. ..."
```

- 출력 형식이 자유로움 → 3B 모델이 캐릭터 말투로 요약하거나, 대화를 그대로 옮기거나, 관련 없는 내용 포함
- `score_importance()`가 키워드 기반이라 요약 텍스트가 비구조화되면 점수 오판 발생
- 잡담 10턴도 무조건 요약 시도 → LLM 호출 낭비, 품질 낮은 요약 VDB 누적

### 3-2. 수정 방향

#### 3-2-A. 구조화 출력 포맷 강제

```python
_SUMMARIZE_SYSTEM = """\
아래 대화에서 사용자에 대한 정보를 추출해. 반드시 아래 형식으로만 답해.
정보가 없는 항목은 '-'로 표시해.

이름: [언급된 경우만]
사건: [중요한 일, 감정적 순간, 약속]
감정: [사용자의 감정 상태]
기타: [취미, 선호도, 반복 화제]
"""
```

`score_importance()`도 이 포맷에 맞게 키워드 검사 — "이름: " 으로 시작하는 줄 감지하면 즉시 1.0.

#### 3-2-B. 요약 조건 강화 — 중요 발화 포함 여부 사전 검사

```python
def should_summarize(dialogue_log: list[dict], trigger_n: int) -> bool:
    """최근 trigger_n턴에 중요 키워드가 하나라도 있을 때만 True."""
    recent = dialogue_log[-(trigger_n * 2):]
    text = " ".join(m["content"] for m in recent)
    important_kw = _NAME_KEYWORDS + _HIGH_KEYWORDS + _MID_KEYWORDS
    return any(kw in text for kw in important_kw)
```

`check_trigger()`와 별도로, `should_summarize()`도 True일 때만 `summarize()` 호출.

#### 3-2-C. `router.py` 호출부 수정

```python
if mode == "chat" and summarizer.check_trigger(self.session, self._trigger_n):
    if summarizer.should_summarize(self.session.dialogue_log, self._trigger_n):
        self._run_summarizer()
    else:
        logger.debug("[router] 요약 조건 미충족 — 생략")
```

### 3-3. 검증

요약 결과가 구조화된 형식으로 나오는지 수동 확인:
```
이름: 이유름
사건: 등대 함께 방문, 파도 소리 좋다고 말함
감정: 긍정적, 편안함
기타: 조용한 장소 선호
```

## 항목 4 — RAG 자연 연결 프롬프트 설계

### 4-1. 현재 문제

`_layer_b()`에 RAG 결과 주입 시 지시문 없음:

```python
parts.append(f"[세계관 참고]\n{rag_text}")
```

3B 모델이 이 블록을 어떻게 사용해야 하는지 모름 → 두 가지 실패 패턴:
- **암송**: RAG 내용을 그대로 대사로 읽어냄 ("여름이 끝날 무렵 등대 아래에...")
- **무시**: RAG 블록을 시스템 노이즈로 처리 후 응답에 반영 안 함

### 4-2. 수정 방향

#### 4-2-A. RAG 히트 여부에 따른 분기 처리

```python
def _layer_b(self, rag_results: list[str] | None = None) -> str:
    ...
    if rag_results:
        rag_text = "\n".join(f"- {r}" for r in rag_results)
        parts.append(
            "[세계관 배경 — 방금 대화와 관련된 정보]\n"
            f"{rag_text}\n"
            "이 정보는 캐릭터가 이미 알고 있는 배경이다. "
            "직접 인용하거나 설명하지 말고, 대화 흐름 안에서 자연스럽게 녹여라."
        )
```

#### 4-2-B. 나레이션 트리거(story/culture)는 현행 유지

`_pending_narration`으로 별도 버블 처리 → LLM 루프 밖에서 처리하는 현재 방식이 올바름.
LLM 컨텍스트에는 `item_title` 힌트만 주입 (현재 구현 확인 필요).

#### 4-2-C. 일반 RAG 히트와 나레이션 트리거 분리 확인

현재 `router.handle_turn()` 흐름:
```
rag_results = self.rag.query(user_input)        # 일반 시맨틱 검색
narration_data = self._check_world_triggers()   # 키워드 기반 나레이션 트리거
```

두 경로가 이미 분리되어 있으나, 동일 텍스트가 양쪽에서 동시에 처리될 경우
LLM 컨텍스트 + 나레이션 버블로 이중 출력될 수 있음 → 확인 및 중복 방지 로직 추가 검토.

### 4-3. 검증

- RAG 히트 있는 질문 입력 → 응답에 자연스럽게 세계관 정보가 녹아 있는지 확인
- 캐릭터가 "등대 아래에 촛불이 켜진다고 알고 있어" 수준이 아닌
  "그 시기가 되면 이 마을이 달라지거든." 수준으로 표현하는지 확인
