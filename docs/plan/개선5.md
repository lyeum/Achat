## 2. 세계관 RAG — 청킹 전략 재설계 ✅ 완료

**목표:** 기존 고정 크기 청킹을 세계관 구조 기반 청킹으로 교체  
**설명:** 현재 `rag/index.py`의 `_chunk_text()`는 텍스트를 고정 크기로 자른다. 세계관의 의미 단위(문화, 장소, 스토리 항목)가 잘려 컨텍스트가 손실된다. 소스 파일 구조를 정의하고 파서를 새로 작성한다.

**소스 파일 구조 (`rag/sources/world/Seaside.md` 기준):**
```
# Seaside              ← 세계관 이름 (최상위 식별자, 모든 chunk 메타에 포함)

## culture
### 항목 제목
내용...

## place
### 항목 제목
내용...

## story
### 챕터/이벤트 제목
트리거 키워드: [키워드1, 키워드2]
내용...
```

**청킹 단위:** `(world_id, section_type, item_title)` 삼중 키로 식별. 기존 `act_id` 기반 청킹 제거.  
**ChromaDB 메타 구조:**
```python
{
  "world_id":    "Seaside",
  "section":     "culture" | "place" | "story",
  "item_title":  "항목 제목",
  "trigger_keywords": "키워드1,키워드2"  # story 전용
}
```

**수정 파일:**
- `rag/index.py` — `index_world()` 청킹 로직 교체, 섹션 파서 추가
- `rag/retrieve.py` — 조회 시 `world_id` + `section` 필터 지원
- `rag/sources/world/Seaside.md` — 기존 파일을 신규 구조로 재작성

## 3. 세계관 트리거 시스템 ✅ 완료

**목표:** culture/place/story 각각 조건에 맞는 나레이션을 세션 내 최초 1회 자동 출력  
**설명:** 세계관 콘텐츠를 단순 RAG 검색 결과가 아닌, 트리거 기반 나레이션으로 제공한다. 모든 트리거는 같은 세션 내 최초 1회만 발동한다.

### 3-1. Story 트리거

- **조건:** 사용자 발화에서 story 항목의 `trigger_keywords`와 코사인 유사도가 임계값 이상일 때
- **출력 형식:** 스토리 전체를 한 번에 출력. `#` 최상위 내용은 펼침, `##` 하위 항목들은 접힘 처리 후 클릭으로 펼치기 가능. 맨 하단에 "진행하기" 버튼으로 선택 루프 탈출
- **상태 관리:** `SessionState.fired_stories: list[str]` — 발동된 story `item_title` 목록. 세션 변경(캐릭터/세계관 변경) 시 초기화

### 3-2. Place 트리거

- **조건:** 장소 변경(`bridge.changeWorld()`의 act_id 변경) 시 해당 장소에 처음 진입할 때
- **출력:** place 항목의 설명 텍스트를 그대로 나레이션으로 출력
- **상태 관리:** `SessionState.visited_places: list[str]` — 방문한 장소 목록. 세션 변경 시 초기화. 장소 변경은 세션 변경이 아님

### 3-3. Culture 트리거

- **조건:** 사용자 발화가 문화/풍습/지역특색 관련 질문으로 판단될 때
- **출력:** 소거법으로 미설명 culture 항목 중 하나를 선택해 설명. 추가 질문이면 다음 항목 설명. 전부 소진 시 "더 이상 없음" 응답. 이미 설명한 항목에 대해 되물으면 재설명 또는 캐릭터가 재해석해서 설명
- **상태 관리:** `SessionState.explained_cultures: list[str]` — 세션 내 임시 상태

### 3-4. 세션 진행 메모리

- 현재 장소/세계관 정보는 세션 내에서 상시 유지되어 대화 컨텍스트에 포함
- 트리거 최초 발동 여부와 무관하게 장소·세계관 기본지식은 RAG로 프롬프트에 지속 주입

**수정 파일:**
- `conversation/session_manager.py` — `SessionState`에 `fired_stories`, `visited_places`, `explained_cultures` 필드 추가
- `narration/narration_hardcoded.py` 또는 신규 `narration/world_trigger.py` — 트리거 판단 + 나레이션 생성
- `ui_ux/qml/main.qml` — story 나레이션 전용 UI 컴포넌트 (접힘/펼침)
- `agent/core.py` — 턴 처리 흐름에 트리거 체크 삽입

## 버그픽스 — 트리거 미작동 + RAM 증가 (발견: 2026-04-09)

개선5 구현 후 실 테스트에서 세계관 트리거가 전혀 발동하지 않고
RAM이 구현 전 대비 +400MB 증가함이 확인됨.

### 진단

| # | 증상 | 원인 |
|---|---|---|
| 1 | story / culture 트리거 미발동 | `world_id` 불일치: ChromaDB에 `"Seaside"` 저장, 세션은 `"seaside_world"` 사용 → `query_by_meta` 항상 0건 반환 |
| 2 | place 트리거 미발동 | `_check_world_triggers`에서 `check_place_trigger` 미호출. `_handle_location` 장소 변경 후 연결 누락 |
| 3 | 세션 시작 시 초기 배경 설명 없음 | 첫 턴 현재 장소 `place` 트리거를 발동시키는 진입점 없음 |
| 4 | RAM +400MB | `query_by_meta`가 `col.get()`(메타 전용)만 필요한데 `_load_ef()`로 bge-m3(~400MB CPU) 불필요하게 로드 |

### 수정 계획

#### BF-1. `Seaside.md` world_id 정합성 수정

**파일:** `rag/sources/world/Seaside.md`

```
# Seaside  →  # seaside_world
```

수정 후 `rag/index.py::index_world()` 재실행하여 ChromaDB 재인덱싱 필요.  
`world_id` 를 `W_sea.yaml`의 `world_id: seaside_world`와 통일한다.

---

#### BF-2. `query_by_meta` 임베딩 로드 제거

**파일:** `rag/retrieve.py`

`query_by_meta`는 `col.get()`만 호출 — 임베딩 불필요.  
`_load_ef()` 대신 embedding_function 없이 컬렉션 조회.

```python
# 변경 전
col = self._client.get_collection(
    name=self._col_name, embedding_function=self._load_ef()
)

# 변경 후
col = self._client.get_collection(name=self._col_name)
```

**효과:** bge-m3 CPU 로드 (~400MB) 방지 → RAM 구현 전 수준 회복.

---

#### BF-3. place 트리거 — `_handle_location` 연결

**파일:** `conversation/core/router.py`

`_handle_location`에서 장소 변경을 감지한 뒤 `check_place_trigger` 호출.  
반환된 나레이션은 `handle_turn`으로 전달해 응답 앞단에 출력.

```python
# _handle_location 반환값 변경: None → narration str | None
def _handle_location(self, user_input: str) -> str | None:
    ...
    # 장소 이동 감지 후 place 트리거 체크
    from narration.world_trigger import check_place_trigger
```

### F-5. 나레이션 출력 경로 오류 — 설계와 구현 불일치 (핵심)

**위치:** `conversation/core/router.py::handle_turn()` 92~96행

**현재 구현:**
```python
narration = self._check_world_triggers(user_input)
if narration:
    messages[-1]["content"] = f"{narration}\n\n{user_input}"  # ← user message에 삽입
response = self.llm.generate(messages, ...)
```
나레이션이 LLM의 **사용자 입력**으로 들어가 LLM이 나레이션 내용에 응답하는 이상한 동작이 발생한다.

**설계 의도 (개선5 항목3):**
> "스토리 전체를 한 번에 출력", "place 항목의 설명 텍스트를 그대로 나레이션으로 출력"
나레이션은 채팅 UI에 **직접 버블로 출력**되어야 한다. LLM 입력이 아님.

**올바른 흐름:**
```
_check_world_triggers() → 나레이션 반환
    ↓
router.handle_turn() → 반환값에 나레이션 포함해 caller(bridge)로 전달
    ↓
bridge._on_response() → 나레이션 있으면 먼저 messageAdded.emit("assistant", narration)
                       → 이후 정상 LLM 응답 emit
```

**수정 방향:**
- `handle_turn()`이 `(response, narration)` 튜플을 반환하도록 변경 (또는 별도 필드)
- `bridge._on_response`가 narration을 먼저 emit한 뒤 LLM 응답을 emit
- LLM은 나레이션 없이 원래 user_input만 처리

**수정 파일:** `conversation/core/router.py`, `ui_ux/bridge.py`, `ui_ux/chat_panel.py`(LLMWorker 반환값)
