## 1. Affection 관리자 조절 + 잠금 ✅

### 구현 결과 (2026-03-29)

| 파일 | 변경 내용 |
|---|---|
| `conversation/core/session.py` | `affection_locked: bool = False`, `affection_lock_value: int \| None = None` 추가 |
| `agent/state.py` | `update_affection()` 상단에 lock 체크 추가 (locked 시 lock_value로 고정) |
| `ui_ux/bridge.py` | `affectionChanged` Signal, `currentAffection` Property, `setAffection()` / `lockAffection()` / `unlockAffection()` 슬롯 추가 |

테스트: `tests/test_dialogue_quality.py::TestAffectionAdminControl` (6개)

### 목표
- 관리자 모드에서 affection 값을 직접 설정 가능
- 특정 수치에 잠금(lock)하여 일시적으로 변화 억제

### 현재 구조

```
ConversationSession.affection  (int, 0~100)
    ↕
agent/state.py::update_affection()  ← 매 턴 router.py에서 호출
    ↕
ui_ux/bridge.py  ← session 상태 읽기/동기화
```

### 수정 대상 파일

| 파일 | 변경 내용 |
|---|---|
| `conversation/core/session.py` | `affection_locked: bool`, `affection_lock_value: int \| None` 필드 추가 |
| `agent/state.py` | `update_affection()` 상단에 lock 체크 추가 |
| `ui_ux/bridge.py` | admin 슬롯 3개 추가, `currentAffection` Property 추가 |
| `conversation/session_manager.py` | lock 상태 영속화 검토 (선택) |

### 설계

**`conversation/core/session.py`**

```python
affection_locked: bool = False
affection_lock_value: int | None = None
```

**`agent/state.py::update_affection()` 수정**

```python
def update_affection(session, mood, character=None) -> int:
    # lock 상태면 lock 값으로 고정 후 반환
    if session.affection_locked and session.affection_lock_value is not None:
        session.affection = session.affection_lock_value
        return session.affection
    # 기존 로직 유지...
```

**`ui_ux/bridge.py` 슬롯 추가**

```python
@Slot(int)
def setAffection(self, value: int):
    """관리자 직접 설정. 0~100 클램핑."""
    self._agent.session.affection = max(0, min(100, value))

@Slot(int)
def lockAffection(self, value: int):
    """특정 수치에 고정."""
    self._agent.session.affection_locked = True
    self._agent.session.affection_lock_value = max(0, min(100, value))
    self._agent.session.affection = self._agent.session.affection_lock_value

@Slot()
def unlockAffection(self):
    """잠금 해제. 이후 정상 update_affection() 동작 재개."""
    self._agent.session.affection_locked = False
    self._agent.session.affection_lock_value = None

@Property(int, notify=affectionChanged)
def currentAffection(self) -> int:
    return self._agent.session.affection if self._agent.session else 30
```

> `affectionChanged` Signal 추가 필요 (moodChanged와 동일 패턴)

## 2. Semantic Score 기반 Affection 게이팅 ✅

### 구현 결과 (2026-03-29)

| 파일 | 변경 내용 |
|---|---|
| `config.py` | 모든 환경에 `"aff_gate_threshold": 0.6` 추가 |
| `conversation/core/router.py` | `_aff_gate` 값 config에서 읽기, `update_affection()` 호출 전 `score_importance()` 게이팅 추가 |

테스트: `tests/test_dialogue_quality.py::TestAffectionSemanticGating` (5개)

### 목표
빈번한 발화(잡담, 단답) 시 매 턴 affection 변화 → 전환점(감정적으로 유의미한 발화)에만 반영

### 현재 구조

```
매 턴 → update_mood() → update_affection()   ← 무조건 호출
```

### 설계

`memory/summarizer.py::score_importance()`가 이미 텍스트 중요도(0.5/0.6/0.85)를 계산한다.
동일 함수를 게이트로 사용한다. 요약 텍스트 대상이지만 단발 발화에도 충분히 동작함.

**`conversation/core/router.py` 수정**

```python
from memory.summarizer import score_importance

# 기존
new_mood = state_mod.update_mood(self.session, user_input, self.character)
state_mod.update_affection(self.session, new_mood, self.character)

# 수정 후
new_mood = state_mod.update_mood(self.session, user_input, self.character)
importance = score_importance(user_input)
if importance >= 0.6:   # mid 이상 발화만 aff 변경
    state_mod.update_affection(self.session, new_mood, self.character)
```

### 게이팅 임계값 기준

| importance | 발화 특성 | affection 변화 |
|---|---|---|
| 0.85 (high) | 이름/약속/갈등/사과/감사 | ✅ 반영 |
| 0.60 (mid) | 감정 표현/취미/날짜 언급 | ✅ 반영 |
| 0.50 (low) | 일상 잡담, 단답, 무키워드 | ❌ 억제 |

> 임계값은 `config.py`에 `aff_gate_threshold: 0.6` 으로 분리해 관리하면 나중에 조정 편함.

### 향후 고도화 (필요 시)
embedding cosine 변화량 기반 게이팅 — 현재 발화 vs 이전 N턴 발화 평균 임베딩 거리를 계산해 "이전과 얼마나 다른 맥락인가"를 게이트로 사용. 매 턴 임베딩 inference 비용 발생. score_importance 방식이 부족할 때 교체.

### 구현 결과 (2026-03-29)

| 파일 | 변경 내용 |
|---|---|
| `tools/prompt_store.py` | 신규: `PromptGuideStore` 클래스 — `prompt_guides` 컬렉션, `save()` / `query()` / `delete()` / `list_models()` |
| `tools/prompt_converter.py` | `__init__(llm, config)` 추가, `execute()`에서 DB 조회 우선 → 크롤링 폴백 → 자동 저장 |

테스트: `tests/test_function_tools.py::TestPromptGuideStore` (7개), `TestPromptConvertFallbackUtils` (10개)

### 목표
매 요청마다 크롤링하는 현재 방식 → 저장된 프롬프트 가이드 DB 조회 방식으로 전환.
웹검색 도구로 가이드를 수집·저장하고, 이후 요청은 DB에서 바로 꺼내 사용.

### DB 저장 위치 (명확화)

현재 `chroma_dev/` 아래 두 컬렉션이 존재한다:

| 컬렉션명 | 분류 | 설명 |
|---|---|---|
| `{char_id}_memory` | 에피소딕 | 캐릭터별 대화 기억 |
| `world_knowledge` | 영구 지식 | 세계관/장소 RAG |

신규 컬렉션 **`prompt_guides`** 를 동일 `chroma_dev/` 경로의 동일 `PersistentClient` 안에 추가한다.
기존 두 컬렉션과 완전히 독립되며, `force=True` 재인덱싱(세계관 재구축)의 영향을 받지 않는다.

```
chroma_dev/
├─ chroma.sqlite3
└─ {uuid}/
   ├─ haru_memory          ← 기존: 에피소딕 기억
   ├─ world_knowledge      ← 기존: 세계관 RAG
   └─ prompt_guides        ← 신규: 프롬프트 가이드 (모델명 → 프롬프트 구조)
```

> `prompt_guides`는 사용자가 명시적으로 저장 명령을 내리지 않는 한 삭제되지 않는다.
> `world_knowledge`의 `force=True` 재인덱싱이 `prompt_guides`를 건드리지 않도록
> `rag/index.py`에서 컬렉션명을 명시적으로 지정하는 구조를 유지해야 한다.

### 저장 스키마

```
{
  id:       "pg_{model_normalized}_{uuid8}"
            예: pg_stable-diffusion-xl_a3f2c1b0
  document: "positive prompt 구조:\n{구조 설명}\n\nnegative prompt:\n{구조 설명}\n\n예시:\n{예시}"
  metadata: {
    "model":    "stable-diffusion-xl",    ← 정규화된 모델명 (소문자, 공백→하이픈)
    "version":  "1.0",                    ← 선택 (모를 경우 "unknown")
    "source":   "user",                   ← "user" | "crawl"
    "saved_at": "2026-03-29T00:00:00Z"
  }
}
```

ChromaDB metadata 타입 제약(str/int/float/bool만 허용)에 따라 모든 필드를 str로 저장.

### 신규 파일

**`tools/prompt_store.py`** — 저장/조회 전담 모듈 (long_term.py와 동일 패턴)

```python
class PromptGuideStore:
    컬렉션명: "prompt_guides"
    임베딩: 기존 bge-m3 공유 (chroma_path config에서 읽음)

    def save(model: str, guide_text: str, source: str = "user") → None
    def query(model: str) → str | None
        ① metadata where {"model": model} 로 exact match 우선
        ② 없으면 semantic search fallback (model명 텍스트 쿼리)
        ③ 둘 다 없으면 None 반환
```

### 수정 파일

**`tools/prompt_converter.py`**

```
현재 흐름:
  model 추출 → 크롤링(_collect_guide) → LLM 변환 → 출력

변경 후 흐름:
  model 추출
      → PromptGuideStore.query(model)
          ↳ 히트: 저장된 가이드 사용 (크롤링 스킵)
          ↳ 미스: 기존 크롤링 fallback (또는 "가이드 없음. 웹검색 후 저장 필요" 안내)
      → LLM 변환 → 출력
```

### 저장 트리거

웹검색 결과를 저장할 때 자연어 명령으로 연결:
- `"stable diffusion xl 프롬프트 가이드 저장해줘"` → `WebSearchTool` + `PromptGuideStore.save()`
- 저장 흐름은 별도 tool 또는 `PromptConverterTool` 내 `save` 파라미터로 처리

### VDB.md 업데이트 필요
`prompt_guides` 컬렉션 항목을 섹션 3 컬렉션 구조 테이블에 추가.

## 7. 웹검색 2-step Prompting ✅

### 구현 결과 (2026-03-29)

| 파일 | 변경 내용 |
|---|---|
| `tools/search/web_search.py` | `WebSearchTool.__init__(llm, character)` 추가, `execute()`에서 LLM 있을 때 2-step 실행 (Step1: `mode="function"` 사실 추출, Step2: `mode="chat"` 캐릭터 어조 적용); LLM 없으면 raw 텍스트 폴백 |
| `agent/core.py` | `WebSearchTool`을 `_STATIC_TOOLS`에서 제거, stub/정상 모드 모두 `WebSearchTool(llm=..., character=...)` 주입 |

테스트: `tests/test_function_tools.py::TestWebSearchTwoStep` (6개)

### 목표
검색 결과를 사실 기반으로 요약한 뒤, 캐릭터 말투로 브리핑하는 2단계 구조.
단일 LLM 호출로 캐릭터 LoRA가 사실 답변 형성을 방해하는 현재 문제를 해결.

### 현재 문제

```
현재: 웹크롤링 → LLM(LoRA ON, 캐릭터 말투 강화) → 응답
→ LoRA가 사실 서술보다 캐릭터 언어 패턴을 우선 적용
→ 검색 내용이 정확하게 전달되지 않거나 hallucination 혼입
```

### LoRA adapter on/off (1차 방안) 채택 불가 이유

현재 배포 방식이 **GGUF 단일 파일**이면, LoRA가 베이스 모델에 병합되어 있어 런타임 on/off 불가.
transformers GPU 환경에서만 `peft_model.disable_adapter_layers()` 사용 가능.
현재 배포 제약(GGUF + 용량)을 유지하는 한 1차 방안 적용 불가.

### 채택 방안: 2-step Prompting (추가 모델 없이 현재 인프라로 구현 가능)

```
Step 1 — 사실 추출 (mode="function")
  system: "아래 검색 결과에서 [질문]에 대한 답변을 추출하라.
           말투나 감정 표현 없이 정보만 정리한다. 불필요한 정보는 제외한다."
  user: f"질문: {user_input}\n\n검색 결과:\n{search_results}"
  → 사실 답변 텍스트 (max_new_tokens=256 제한)

Step 2 — 캐릭터 브리핑 (mode="chat")
  system: [기존 캐릭터 시스템 프롬프트 Layer A]
  user: f"다음 정보를 {character.name}의 말투로 브리핑해줘.
         내용 자체의 의미는 바꾸지 않는다:\n\n{step1_result}"
  → 캐릭터 말투가 적용된 최종 응답
```

### 프롬프트 레이어 구조

```
[검색 + 크롤링]
    ↓
[Step 1: 사실 추출 LLM] ← mode="function" (LoRA 캐릭터 영향 최소화)
    ↓
[Step 2: 캐릭터 말투 오버레이 LLM] ← mode="chat" (캐릭터 LoRA 활용)
    ↓
[최종 응답] — 내용은 Step 1 기반, 말투는 캐릭터
```

### 수정 대상 파일

| 파일 | 변경 내용 |
|---|---|
| `tools/search/web_search.py` | `execute()` 내부에 2-step 흐름 추가 |
| `agent/core.py` | web_search 도구에 LLM + character 전달 확인 (현재 LLM 주입 여부 확인 필요) |

### 구현 시 주의사항

- Step 1 `max_new_tokens`는 256 이하로 제한 → 응답 시간 관리
- Step 2는 기존 캐릭터 프롬프트(`PromptBuilder.assemble()` Layer A)를 재사용
- "캐릭터가 모른다고 해야 할 상황"(검색 결과 없음) 처리: Step 1에서 "정보 없음" 반환 → Step 2 스킵, 캐릭터 말투로 "모른다" 응답 생성
