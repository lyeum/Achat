### 6-2. 유형별 근본 원인

#### ① 반복 루프

**원인:**
- `llm_client.py` `_generate_transformers()` chat mode `repetition_penalty=1.1` — 3B 모델에게 충분히 강하지 않음
- `_generate_llama()`는 `repeat_penalty`를 아예 넘기지 않음 → llama.cpp 기본값 1.0 사용
- 짧은 응답 패턴("...여기 있어")이 한번 활성화되면 이후 모든 생성에서 동일 토큰 시퀀스를 반복 선택

**해결 방안:**
```python
# llm_client.py _generate_llama() 수정
output = self._model.create_chat_completion(
    messages=messages,
    max_tokens=max_tokens,
    stream=stream,
    repeat_penalty=1.15,          # ← 추가
    repeat_last_n=64,             # ← 최근 64 토큰 범위에서 반복 억제
)

# _generate_transformers() chat mode 수정
gen_kwargs = dict(do_sample=True, temperature=0.8, repetition_penalty=1.15)  # 1.1 → 1.15
```

**추가 보완:** `no_repeat_ngram_size=4` 추가 검토 (4-gram 이상 반복 원천 차단)

**해결 방안 (즉시):** YAML rules에 이미 "한국어가 아닌 다른 언어의 단어를 문장 중간에 섞지 않는다" 포함 — ② 번 rules 포맷 수정(줄바꿈 목록화)으로 인식률 향상 기대

**해결 방안 (학습):**
- 학습 데이터 전수 검사 후 영어 혼입 예시 완전 제거
- feedback_neg의 영어 혼입 사례를 `correction_graceful.jsonl`에 추가하되, 잘못된 응답은 제외하고 올바른 응답만 수록

#### ⑤ 서술자/나레이터 혼입

**원인 1:** 학습 데이터에서 `*행동*` 포맷이 불일치 — `*난처한 투로 대답한*`(과거형·3인칭) vs `*조용히 네 쪽을 보며*`(현재형·행동) 혼용

**해결 방안 (학습):** 행동 표현 포맷 통일 기준 확립

#### ⑦ 맥락 끊김

**원인:** `prompt_build.py` `_layer_d()` 토큰 예산 `BUDGET["layer_d"] = 300`이 부족
- 한국어 300토큰 ≈ 600자 ≈ 약 3~4턴 → 대화가 깊어지면 예산 초과 후 `short_buf[-2:]` (1턴)으로 축소
- 이전 발화를 잃어버리면 캐릭터 톤이 리셋됨

**해결 방안:**
```python
# prompt_build.py BUDGET 수정
BUDGET = {
    "layer_a": 300,
    "layer_b": 200,
    "layer_c": 150,
    "layer_d": 450,    # 300 → 450 (약 2턴 추가 보장)
    "generation": 512,
}

# _layer_d() 최소 턴 보장 강화
def _layer_d(self, short_buf: list[dict]) -> list[dict]:
    for n_turns in (5, 4, 3):   # 2 → 3 으로 하한 상향
        sliced = short_buf[-(n_turns * 2):]
        total = sum(self._count(m["content"]) for m in sliced)
        if total <= BUDGET["layer_d"]:
            return sliced
    return short_buf[-6:]       # 최소 3턴 보장
```

### 6-5. 세계관 정보 재가공 허용 (방향 전환)

### 6-6. 캐릭터 자기 발화 기억 (신규 기능 설계)

> 작성일: 2026-04-20

#### 문제 정의

현재 메모리 구조의 구조적 공백:

```
summarizer._SUMMARIZE_SYSTEM:
  "아래 대화에서 사용자에 대한 정보를 추출해."
  → 캐릭터가 한 말은 추출 대상 없음
```

`dialogue_log`에 `assistant` 턴으로 존재하다가 layer_d 버퍼 트리밍 시 **완전히 소멸**. VDB에도 저장되지 않음.

**결과:** "같이 가줄게", "기억하고 있어", "그거 말하는 거야?" 등 캐릭터 자신의 약속·언급이 10턴 지나면 없었던 일이 됨.

#### 아키텍처 설계 (3계층)

```
[실시간 — 세션 내]
  router.py: assistant 응답에서 약속/언급 패턴 감지
          → session.character_notes: list[str] 추가
          → prompt_build.py Layer E로 매 턴 시스템 주입

[단기 — 버퍼]
  short_term.get_recent() : 기존 유지 (5턴 윈도우)

[장기 — 세션 간 VDB]
  summarizer: 캐릭터 발화 필드 추가
          → character_promise: [캐릭터가 한 약속/제안]
          → VDB 저장 → 다음 세션에서도 검색 가능
```

#### 구현 상세

**① ConversationSession — character_notes 필드 추가**
```python
# conversation/core/session.py
@dataclass
class ConversationSession:
    ...
    character_notes: list[str] = field(default_factory=list)
    # 세션 내 캐릭터 발화 중 약속·언급·감정 표현 핵심 내용
```

**② router.py — 약속 패턴 감지 및 notes 추적**
```python
# router.py handle_turn() — LLM 응답 생성 직후
response = self.llm.generate(messages, stream=stream)

# 캐릭터 발화에서 약속·언급 패턴 감지
_PROMISE_PATTERNS = [
    "~할게", "~하자", "~데려갈게", "~기억할게", "~기억하고 있어",
    "~있을게", "~같이", "~알려줄게", "~말할게",
]
# 간단 구현: 패턴 포함 시 핵심 문장 추출 후 notes에 추가
note = _extract_character_note(response)
if note:
    self.session.character_notes.append(note)
    # 최대 10개까지만 유지 (FIFO)
    self.session.character_notes = self.session.character_notes[-10:]
```

**③ prompt_build.py — Layer E (캐릭터 자기 메모)**
```python
# PromptBuilder.assemble() 에 layer_e 추가
def assemble(self, ...) -> list[dict]:
    system_parts = [self._layer_a(...), layer_b]
    if vdb_results:
        system_parts.append(self._layer_c(vdb_results))
    if self.session.character_notes:         # ← 추가
        system_parts.append(self._layer_e()) # ← 추가
    if recent_ops:
        system_parts.append(self._layer_f(recent_ops))
    ...

def _layer_e(self) -> str:
    notes = self.session.character_notes
    if not notes:
        return ""
    items = "\n".join(f"- {n}" for n in notes)
    return (
        "[자신이 했던 말]\n"
        + items
        + "\n위 내용을 인지하고, 대화 흐름에서 자연스럽게 연결한다."
    )
```

**④ summarizer.py — 캐릭터 발화 필드 추가 (세션 간 영속화)**
```python
_SUMMARIZE_SYSTEM = """\
아래 대화에서 정보를 추출해. 반드시 아래 형식으로만 답해.
정보가 없는 항목은 '-'로 표시해.

이름: [사용자 이름]
사건: [중요한 사건, 감정적 순간]
감정: [사용자가 드러낸 감정 상태]
기타: [사용자 취미, 선호도]
캐릭터_약속: [캐릭터가 한 약속·제안·언급 — 없으면 -]"""
```

`score_importance()`에서 `캐릭터_약속` 비어있지 않으면 최소 0.4 보장.

#### 우선순위

| 구현 | 효과 | 난이도 |
|---|---|---|
| ①②③ session.character_notes + Layer E | 세션 내 발화 기억 즉각 효과 | 낮음 |
| ④ summarizer 캐릭터_약속 필드 | 세션 재시작 후에도 기억 유지 | 낮음 |
| `_extract_character_note()` 정교화 | 약속 추출 정확도 향상 | 중간 |

①②③은 세션 객체와 프롬프트 빌더만 수정하면 되므로 재학습 없이 즉시 적용 가능.  
④는 다음 summarizer 동작 시점부터 자동으로 캐릭터 발화도 VDB에 저장됨.

> **설계 업데이트:** Layer E의 최종 구조는 7-4절에서 확정됨.
> 위 6-6의 `_layer_e()` 코드는 초안이며, session_context + character_notes를 통합한 7-4 설계를 따른다.

## 7. 단기/장기 메모리 구조 개선

> 작성일: 2026-04-21

### 7-1. 현재 구조의 문제

#### 단기/장기 기준 불일치

| 기준 | 설정 값 | 실제 동작 |
|---|---|---|
| `config.short_term_n = 5` | 5턴 | **실제 미사용** — `get_recent(n=None)` 으로 호출, 전체 반환 |
| `BUDGET["layer_d"] = 300` | 300 토큰 | 토큰 초과 시 5→3→2→1턴으로 축소 |
| `memory_trigger_n = 5` | 5턴마다 | `should_summarize()` 키워드 조건부 저장 |

`short_term_n`과 실제 단기 창 크기가 다른 곳에서 다른 기준으로 결정됨.

#### 단기→장기 사이 공백 (가장 큰 문제)

```
단기 버퍼 (최근 ~3턴, 토큰 기준)
         ↓ 밀려남
    [공백 — 유실 위험]
         ↓ 키워드 조건 통과 시
장기 VDB (세션 간 영속)
```

버퍼에서 밀려난 턴이 키워드 조건을 통과하지 못하면 영구 소멸.
30턴 대화에서 버퍼 밖 25턴의 내용이 VDB에 있는지 없는지가 불확실.

#### summarizer가 `return response` 전에 실행됨 (latency 문제)

```python
# router.py
self._run_summarizer()   # ← 라인 143: 요약 LLM 호출 (blocking)
...
return response          # ← 라인 150: 사용자에게 응답 반환
```

매 5턴마다 사용자 체감 응답 시간에 LLM 요약 시간이 추가됨.

### 7-2. 개선 구조 (3계층)

```
[단기] short_buf — 최근 5턴 고정 (turn 수 기준, 토큰 예산 아님)
         ↓ 버퍼에서 밀려난 턴 → 원문 텍스트 그대로 누적
[중기] session.session_context — 세션 내 전체 흐름 압축 보관
         ↓ 세션 종료 또는 score ≥ 0.65 조건
[장기] VDB — 이름/약속/감정적 사건만 선별 저장
```

**핵심 변경:**
- 중기 레이어 추가 (`session.session_context`)
- 단기 기준을 turn 수로 명확히 고정
- VDB 저장 기준을 올려서 품질 확보

### 7-3. session_context 구현 방식

**LLM 호출 없이 구현** — evicted 턴을 원문 텍스트로 누적 (토큰 캡 적용)

```python
# conversation/core/session.py
@dataclass
class ConversationSession:
    ...
    session_context: str = ""
    # 단기 버퍼에서 밀려난 이전 턴의 원문 압축본
    # 형식: "사용자: ...\n캐릭터: ...\n사용자: ..." (최대 300 토큰 분량)
```

```python
# memory/short_term.py — eviction 시 session_context 갱신
SHORT_TERM_N = 5   # 단기 버퍼 턴 수 (토큰 기준 아님, 고정)
CONTEXT_MAX_CHARS = 600  # session_context 최대 길이 (≈ 300 토큰)

def evict_to_context(session: ConversationSession) -> None:
    """dialogue_log에서 SHORT_TERM_N 초과 턴을 session_context로 이동."""
    log = session.dialogue_log
    if len(log) <= SHORT_TERM_N * 2:
        return
    evicted = log[:-(SHORT_TERM_N * 2)]
    lines = [
        f"{'사용자' if m['role'] == 'user' else '캐릭터'}: {m['content']}"
        for m in evicted
    ]
    raw = "\n".join(lines)
    # 토큰 캡: 뒤에서부터 잘라서 최신 내용 우선 보존
    if len(raw) > CONTEXT_MAX_CHARS:
        raw = "...\n" + raw[-CONTEXT_MAX_CHARS:]
    session.session_context = raw
```

**LLM 호출이 필요 없는 이유:**  
session_context의 목적은 "이번 세션에서 어떤 말이 오갔는지"를 모델에게 알려주는 것.  
요약보다 원문이 맥락 연결에 더 정확하고, 어차피 Layer E 토큰 예산 내에서 잘린다.

### 7-4. prompt_build.py — Layer E (session_context 주입)

```python
# prompt_build.py BUDGET 수정
BUDGET = {
    "layer_a": 300,
    "layer_b": 200,
    "layer_c": 150,
    "layer_d": 450,    # 6-3 M항목 채택 (300→450, 맥락 끊김 완화)
    "layer_e": 300,    # session_context 전용 예산 (신규)
    "generation": 512,
}

# _layer_e() 신규
def _layer_e(self) -> str:
    ctx = getattr(self.session, "session_context", "")
    notes = getattr(self.session, "character_notes", [])
    parts = []
    if ctx:
        parts.append("[이전 대화 흐름]\n" + ctx)
    if notes:
        parts.append("[자신이 했던 말]\n" + "\n".join(f"- {n}" for n in notes))
    if not parts:
        return ""
    return "\n\n".join(parts) + "\n위 내용을 인지하고 대화 흐름에서 자연스럽게 연결한다."
```

session_context(이전 흐름)와 character_notes(캐릭터 약속)를 하나의 Layer E로 통합.

### 7-5. summarizer 개선 (VDB 품질 + latency 해소)

#### 문제 요약

| 문제 | 내용 |
|---|---|
| latency 차단 | `_run_summarizer()`가 `return response` 전에 실행 — 매 5턴 체감 지연 |
| score 기본값 0.5 | 빈 요약도 threshold(0.5) 통과 → VDB 오염 |
| 구조화 포맷 | "이름: -\n사건: -\n감정: -\n기타: -" 임베딩에 노이즈 |
| 필드 혼재 | `사건: [사건, 감정적 순간, 약속]` — 3가지를 한 필드에 |
| 캐릭터 발화 누락 | "사용자에 대한 정보만 추출" — 캐릭터 약속/제안 미수집 |

#### latency 해소 — 백그라운드 스레드로 이동

```python
# router.py _run_summarizer() 수정
import threading

def _run_summarizer(self) -> None:
    # 세션 상태 스냅샷 (스레드 안전)
    log_snapshot = list(self.session.dialogue_log)
    session_snap = self.session

    def _bg():
        summary = summarizer.summarize(log_snapshot, self.llm, self._trigger_n)
        score   = summarizer.score_importance(summary)
        summarizer.write_to_vdb(
            summary, score, session_snap, self.long_term,
            self.character, trigger_n=self._trigger_n,
        )

    threading.Thread(target=_bg, daemon=True).start()
```

단, `llm._generate_lock`이 serialized이므로 주 대화 LLM 호출과 배경 요약이 겹치면 어느 쪽이든 락 대기가 발생함. 5턴 트리거이므로 실제로는 다음 턴 요청 전에 완료될 가능성이 높지만, **동시 메시지 전송 시 대기 가능성은 남음.**

> 근본 해결은 summarizer용 별도 LLM 인스턴스 또는 rule-based 추출로 LLM 의존 제거.  
> 단기적으로는 백그라운드 스레드로 체감 latency 제거만 해도 충분.

#### 역할 분리 2-call 방식은 오버스펙

VDB 요약과 session_context를 각각 별도 LLM 호출로 생성하는 방식은 다음 이유로 채택하지 않는다.

- **RAM**: 같은 모델 인스턴스를 재사용하므로 추가 메모리 없음. 단, `_generate_lock` serialized 구조에서 2회 호출 = 2× 대기 시간.
- **latency**: 5턴마다 blocking 지연이 2배. 이미 1회도 문제인 상황에서 악화.
- **session_context는 LLM 불필요**: 목적이 "이번 세션에서 어떤 말이 오갔는지"이므로 원문 텍스트 자체가 요약보다 정확하고 비용이 0. evicted 턴을 600자 캡으로 잘라 누적하면 충분.
- **VDB 요약**: 기존 1회 호출에서 프롬프트 개선 + `_to_prose()` 후처리 추가로 해결. 추가 호출 불필요.

#### score 기준 정상화

```python
# summarizer.py
def score_importance(summary: str) -> float:
    score = 0.0        # 기본값 0.5 → 0.0 (키워드 없으면 저장 안 함)
    ...

def write_to_vdb(...):
    if score < 0.65:   # threshold 0.5 → 0.65 (잡담 차단)
        return False
```

#### 출력 형식 자연어화 (임베딩 품질 향상)

```python
_SUMMARIZE_SYSTEM = """\
아래 대화에서 중요한 정보를 추출해. 반드시 아래 형식으로만 답해.
정보가 없는 항목은 '-'로 표시해. 형식 외 다른 말은 쓰지 마.

고유정보: [사용자 이름·나이·직업 등 고유 식별 정보 — 없으면 -]
사건: [이번 대화에서 일어난 일, 결정, 약속 — 없으면 -]
캐릭터_약속: [캐릭터가 한 약속·제안·선언 — 없으면 -]
감정: [사용자가 드러낸 감정 상태 — 없으면 -]
선호도: [사용자 취미·좋아하는 것·싫어하는 것 — 없으면 -]"""
```

단, VDB에 저장할 때는 구조화 포맷 그대로 저장하지 않고 **자연어 문장으로 변환 후 저장**:

```python
def _to_prose(summary: str) -> str:
    """구조화 요약 → 자연어 서술 변환. 임베딩 품질 향상 목적."""
    lines = {}
    for line in summary.splitlines():
        if ": " in line:
            k, v = line.split(": ", 1)
            if v.strip() and v.strip() != "-":
                lines[k.strip()] = v.strip()
    parts = []
    if "고유정보" in lines:
        parts.append(lines["고유정보"] + ".")
    if "사건" in lines:
        parts.append(lines["사건"] + ".")
    if "캐릭터_약속" in lines:
        parts.append(f"캐릭터는 {lines['캐릭터_약속']}.")
    if "감정" in lines:
        parts.append(f"사용자는 {lines['감정']} 감정 상태였다.")
    if "선호도" in lines:
        parts.append(lines["선호도"] + ".")
    return " ".join(parts) if parts else summary
```

## 8. 컨텍스트 관리 전략

### 8-0. 현황 문제

| 구조 | 현재 동작 | 문제 |
|---|---|---|
| `dialogue_log` | 세션 내 무한 증가 | RAM 증가, Layer D 버짓 초과 시 오래된 턴 소실 |
| VDB | 세션 종료 후에도 영구 누적 | 중요도 낮은 항목이 검색을 오염시킴 |
| session_context | 필드 없음 | 단기→장기 사이 "공백 구간" 존재 |
| 수명 관리 | TTL/quota 없음 | 캐릭터별 VDB 항목 제한 없음 |

---

### 8-1. 세션 라이프사이클

```
세션 시작
  ├─ VDB 주기 정리 (score < 0.65 항목 중 TTL 만료된 것 삭제)
  ├─ session_context = "" (빈 상태에서 시작)
  └─ character_notes = [] (promise 트래킹 초기화)

대화 중 (매 턴)
  ├─ on_turn() → Layer D 버짓 계산
  ├─ 버짓 초과된 오래된 턴 → evict_to_context() → session_context 추가
  └─ promise 패턴 감지 → character_notes 추가

대화 중 (매 5턴)
  └─ _run_summarizer() 백그라운드 실행 (non-blocking)
       └─ score ≥ 0.65 → write_to_vdb()

세션 종료 (flush_remaining)
  ├─ 잔여 버퍼 강제 요약 + VDB 저장
  ├─ session_context → session_state.json 스냅샷 저장
  └─ dialogue_log RAM 해제
```

---

### 8-2. Flush 트리거 정책

| 트리거 | 대상 | 조건 |
|---|---|---|
| 매 턴 | session_context | SHORT_TERM_N(5턴) 초과 시 오래된 턴을 raw text로 evict |
| 매 5턴 | VDB | 백그라운드 스레드에서 요약 → 저장 |
| 세션 종료 | VDB + session_context | `flush_remaining()` 호출 시 강제 실행 |
| 중요 이벤트 | VDB | affection 변화 ±5 이상, 고백/배신 이벤트 → 즉시 저장 |

**session_context 상세:**
- `dialogue_log` 에서 SHORT_TERM_N(5턴) 초과 시 밀려난 턴을 600자 한도로 누적
- 초과 시 앞부분부터 잘라내기 (FIFO)
- LLM 호출 없음 — raw text 그대로 유지

---

### 8-3. VDB 보존 정책 (장기 메모리 수명)

```
importance score 기준:

  score ≥ 0.90  →  영구 보존 (TTL 없음)
                    "고백", "배신", "첫 만남" 등 서사 전환점
  
  0.65 ≤ score < 0.90  →  TTL 30일
                           일상 감정 교환, 선호도, 약속 등
  
  score < 0.65  →  저장 안 함 (write_to_vdb에서 즉시 반환)
```

**캐릭터별 quota:**
- VDB 항목 최대 200개/캐릭터
- 초과 시 score 낮은 항목부터 자동 삭제 (세션 시작 시 정리)

**중복 방지:**
- 저장 전 cosine similarity 검사 (threshold 0.85 이상이면 기존 항목 업데이트로 처리)
- `conversation_logger.py`의 Jaccard 방식과 별개로 VDB 레이어에서 추가 dedup

---

### 8-4. 저장 비용 요약

| 저장소 | 항목당 크기 | 최대 항목 수 | 예상 상한 |
|---|---|---|---|
| session_context | ~600자 (raw text) | 1 (세션 내) | ~1.2KB/세션 |
| character_notes | ~50자/약속 | 10개 (session) | ~500B/세션 |
| VDB (장기) | ~200자 (prose) | 200개/캐릭터 | ~40KB/캐릭터 |
| dialogue_log | 가변 (턴당 ~100자) | Layer D trim 시 evict | RAM only |

VDB는 ChromaDB 기준 캐릭터 1명당 40KB 이하 — 무시 가능한 수준.
세션 종료 후 RAM 해제 시 실질 영구 저장은 VDB만 해당.
