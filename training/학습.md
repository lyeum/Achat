# LoRA 학습 매뉴얼

> 환경: Ubuntu 24.04 WSL2 / RTX 5060 Ti (VRAM 8GB / RAM 8GB) / CUDA 12.8
> 모델: Qwen2.5-3B-Instruct
> 방식: bfloat16 풀 파라미터 + LoRA (BitsAndBytes 4-bit 미사용 — Blackwell SM 10.x 미지원)

---

## 0. 현재 학습 방식 구조

### 0-1. LoRA가 가중치를 다루는 방식

기존 베이스 모델(Qwen2.5-3B)의 **모든 파라미터는 완전히 동결(freeze)** 된다.
새로운 가중치를 직접 추가하는 대신, 각 대상 레이어에 **저랭크 행렬 쌍 A·B를 삽입**하여 그것만 학습한다.

```
[기존 레이어 — 완전 동결]
    W_original (예: q_proj, 2048×2048)
    requires_grad = False  ← 절대 바뀌지 않음

[LoRA 삽입 — 이것만 학습]
    W_A  shape: (r, in_dim)    r=16 이면 (16, 2048)
    W_B  shape: (out_dim, r)   r=16 이면 (2048, 16)
    requires_grad = True

[추론 시 실제 적용값]
    W_effective = W_original + (alpha/r) × (W_B @ W_A)
                              ↑ 스케일링: alpha=32, r=16 → ×2.0
```

`W_A`는 랜덤 가우시안으로 초기화, `W_B`는 0으로 초기화 → **학습 시작 시 LoRA 기여분은 0**이라 안정적.

---

### 0-2. 현재 학습 대상 레이어 (target_modules)

Transformer 블록의 Attention 4개 + FFN 3개, **전 28개 레이어에 걸쳐** 모두 적용.

| 모듈 | 역할 | 위치 |
|---|---|---|
| `q_proj` | Query 투영 | Attention |
| `k_proj` | Key 투영 | Attention |
| `v_proj` | Value 투영 | Attention |
| `o_proj` | Output 투영 | Attention |
| `gate_proj` | FFN 게이트 (SiLU) | Feed-Forward |
| `up_proj` | FFN 업 투영 | Feed-Forward |
| `down_proj` | FFN 다운 투영 | Feed-Forward |

Bias는 모두 동결(`bias="none"`). Embedding / LayerNorm 도 동결.

---

### 0-3. 파라미터 수

```
전체 모델:    3,090,000,000 파라미터  (동결)
LoRA 추가분:     ~30,000,000 파라미터  (학습)  ← 전체의 약 1%

계산:
  레이어당 LoRA 파라미터 = 7개 모듈 × 2 × (r × dim)
  예) r=16, dim=2048 → 7 × 2 × 16 × 2048 ≈ 458,752
  28 레이어 × 458,752 ≈ 12.8M  (단방향 기준 — 실제는 양방향 포함 ~30M)
```

---

### 0-4. 정밀도 및 메모리 관리

| 항목 | 설정 | 이유 |
|---|---|---|
| 모델 로드 정밀도 | **bfloat16** | RTX 5060 (Blackwell) 4-bit 양자화 미지원 |
| LoRA 가중치 정밀도 | bfloat16 (자동) | 모델과 동일 dtype 따라감 |
| Gradient 계산 | float32 누적 | 기울기 소실 방지 |
| Gradient Checkpointing | **활성** | Activation을 저장 않고 backward 시 재계산 → VRAM ~30% 절약 |
| `use_cache = False` | 강제 비활성 | KV 캐시와 gradient checkpointing 충돌 방지 |

### 0-5. Loss 계산 범위 (현재 방식의 한계)

```
[토크나이징 결과]
input_ids: [<system>...하루다</s>][<user>뭐해</s>][<assistant>그냥</s>]
labels:    [  동일  ............][  동일  .......][   동일  ........]
                ↑                      ↑                  ↑
            loss 계산됨           loss 계산됨         loss 계산됨
```

현재는 `labels = input_ids` 로 **전체 토큰에 대해 loss를 계산**한다.
시스템 프롬프트·유저 턴까지 loss에 포함되므로 모델이 "무엇을 예측해야 하는가"가 다소 모호해진다.

> **v7부터 적용**: assistant 토큰 구간만 loss 계산 (시스템·유저 구간 labels → -100 마스킹)
> → 응답 품질 직접 최적화, 과적합 억제 효과. `tokenize_dataset._mask_labels()` 구현 완료.

---

### 0-6. 옵티마이저 및 스케줄러

```
옵티마이저: AdamW
  - lr: 2e-4 (초기값)
  - weight_decay: 0.01 (L2 정규화)
  - beta1=0.9, beta2=0.999 (기본값)

스케줄러: Cosine Decay
  - warmup: 전체 스텝의 5% (loss 안정화 후 감쇠 시작)
  - 최종 lr → 0 으로 수렴

유효 배치:
  batch_size(1) × grad_accum(8) = 8
  → 8샘플 분의 gradient를 누적 후 1회 업데이트
```

---

### 0-7. Best Model 선택 방식

```
eval_split > 0 일 때:
  save_steps(100)마다 eval loss 계산
  → 가장 낮은 eval loss 시점의 가중치를 자동 선택
  → 학습 완료 후 adapter/ 에 best 가중치 저장

eval_split = 0 일 때:
  best 자동 선택 불가 → 마지막 epoch 가중치 저장
  (v3 실측: epoch 3~4가 best, 이후 eval loss 반등)
```

## 1. 학습 루프 내부 흐름

```
[데이터 로드]
training/data/**/*.jsonl (또는 data/lora/)
    — `_excluded/` 하위 파일은 자동 제외 (dataset.py load_jsonl_files)
    │
    ▼ load_training_data()
HuggingFace Dataset 객체 (text 필드)
    │
    ▼ tokenize_dataset()   — truncation=True, max_length=512
    │                        labels = _mask_labels(input_ids)
    │                        → assistant 응답 구간만 loss 계산 (v7~)
    │                        → 시스템·유저 토큰 구간은 labels=-100 마스킹
    │
    ▼ train_test_split()   — eval_split > 0 이면 90% train / 10% eval 분리
    │                        eval_split = 0 이면 전체 train, eval 없음
    │
[모델 로드]
AutoModelForCausalLM (bfloat16, device_map="auto")
    │
    ▼ get_peft_model()     — LoRA 어댑터 주입
    │                        target: q/k/v/o/gate/up/down proj (전 레이어)
    │                        trainable params: ~30M / 전체 3.1B
    │
[학습 루프 — HuggingFace Trainer]
    │
    ├─ 매 스텝:
    │   forward → loss 계산 → backward → gradient accumulation(8스텝)
    │   → optimizer step (AdamW + cosine LR decay)
    │
    ├─ logging_steps마다:  train loss / grad_norm / lr / epoch 출력
    │
    ├─ save_steps(100)마다:
    │   eval_split > 0 → eval loss 계산 → best이면 checkpoint 교체
    │   eval_split = 0 → checkpoint만 저장 (eval 없음)
    │
    └─ 학습 완료:
        load_best_model_at_end=True (eval 있을 때) → best checkpoint 복원
        model.save_pretrained() → output/{run_name}/adapter/ 저장
        checkpoint-* 삭제 (가중치·그래프·로그 제외)
        │
        ▼  --skip_eval 없을 때만
        VRAM 완전 해제
            trainer.model=None → del trainer / model / tokenizer
            gc.collect() → torch.cuda.empty_cache() + synchronize()
            여유 VRAM 로그 출력
        │
        ▼  subprocess (독립 프로세스, 각각 순차 실행)
        training/eval/ai_tell_checker.py --adapter {adapter_path}
        training/eval/memory_test.py     --adapter {adapter_path}
```

**총 스텝 수 계산:**
```
총 스텝 = (샘플 수 × 0.9) ÷ (batch_size × grad_accum) × epochs
예 | 2150 × 0.9 ÷ (1 × 8) × 3 ≈ 726 스텝   (v10 기준 — 2,150건, 3 epoch, 권장)
예 | 2401 × 0.9 ÷ (1 × 8) × 3 ≈ 810 스텝   (v9 기준 — 2,401건, 3 epoch)
예 | 2401 × 0.9 ÷ (1 × 8) × 5 ≈ 1351 스텝  (v8 기준 — 2,401건, 5 epoch, 과적합 발생)
```

## 3. 실시간 모니터링

tqdm 출력은 `\r` 덮어쓰기 방식이라 로그 파일에서 grep이 안 됨.
**checkpoint의 trainer_state.json을 직접 읽는 방식** 사용 (100 스텝마다 갱신).

### 학습 속도 패턴 (정상 동작)

tqdm의 `s/it` 값이 학습 도중 급등했다가 점점 줄어드는 패턴이 반복된다. 이는 버그가 아니라
**eval 실행 시 gradient_checkpointing이 일시 비활성화**되기 때문이다.

```
정상 패턴 예시 (v9 기준):
  초반     5~6s/it   ← CUDA warm-up 구간, 짧은 시퀀스 배치
  급등    90s/it    ← eval 실행 (save_steps=100 도달)
             gradient_checkpointing=False로 전환 → full activation VRAM 점유
  회복    66→50→38→24→17s/it  ← train 모드 재전환 후 점진적 복구
  안정    12~17s/it  ← 이후 정상 속도
```

- eval이 끝날 때마다 이 패턴이 반복된다 (`save_steps=100`마다 1회).
- EWC 적용 시 패널티 계산이 매 스텝에 추가되어 비EWC 학습보다 전반적으로 느리다.
  - v8 (EWC 없음): ~9s/it / v9 (EWC λ=500): ~14s/it

```bash
# 현재 스텝당 속도 추이 확인
grep -o '[0-9]*\.[0-9]*s/it' output/LoRA_vN/train.log | tail -20  #FIX# 버전명
```

### loss 그래프 (자동 생성)

학습 중 **epoch 종료마다** + **전체 학습 완료 시** 자동으로 PNG 저장됨:

```
output/LoRA_vN/            #FIX# 버전명
├─ loss_curve_epoch01.png   ← epoch 1 종료 시
├─ loss_curve_epoch02.png   ← epoch 2 종료 시
├─ ...

```

- train loss: 파란 실선
- eval loss: 빨간 실선 + 원 마커 (eval_split > 0 일 때만)
- best eval 지점: 점선 수직선 표시

## 4. 인자 전체 목록

| 인자 | 기본값 | 설명 |
|---|---|---|
| `--model` | `Qwen/Qwen2.5-3B-Instruct` | HuggingFace 모델명 |
| `--data_dir` | `training/data` | 학습 데이터 루트 디렉토리 |
| `--output_dir` | `output/LoRA_v7` | 체크포인트 및 어댑터 저장 경로 |
| `--subset` | `None` (전체) | `conversation` / `function` / `None`(전체 혼합) |
| `--epochs` | `3` | 학습 에폭 수 |
| `--batch_size` | `1` | 디바이스당 배치 크기 |
| `--grad_accum` | `8` | Gradient accumulation 스텝 수 (유효 배치 = batch×grad_accum) |
| `--lr` | `2e-4` | 초기 학습률 (cosine decay 적용) |
| `--max_length` | `512` | 최대 시퀀스 길이 (초과 시 잘림) |
| `--lora_r` | `16` | LoRA rank (높을수록 파라미터 많음, 표현력 증가) |
| `--lora_alpha` | `32` | LoRA alpha (스케일링 = alpha/r, 통상 r의 2배) |
| `--lora_dropout` | `0.05` | LoRA dropout 비율 (과적합 억제 시 `0.1` 권장) |
| `--weight_decay` | `0.0` | L2 정규화 강도 (과적합 억제 시 `0.01` 권장) |
| `--eval_split` | `0.1` | 검증셋 비율 (`0` 이면 eval 완전 비활성) |
| `--max_samples` | `-1` (전체) | 학습에 사용할 최대 샘플 수 (shuffle 후 선택) |
| `--save_steps` | `100` | 체크포인트 저장 및 eval 실행 주기 (스텝 단위) |
| `--logging_steps` | `10` | train loss 출력 주기 (스텝 단위) |
| `--max_steps` | `-1` (에폭 기준) | 양수 설정 시 해당 스텝 수에서 강제 종료 |
| `--no_save` | `False` | 학습 후 어댑터 저장 건너뜀 (파이프라인 테스트용, 자동 평가도 실행 안 함) |
| `--skip_eval` | `False` | 학습 완료 후 자동 평가(`training/eval/`) 건너뜀 (VRAM 해제도 생략됨) |
| `--ewc_fisher` | `None` | `fisher.pt` 경로 — `--ewc_lambda > 0` 이면 필수 |
| `--ewc_ref_params` | `None` | `ref_params.pt` 경로 — `--ewc_lambda > 0` 이면 필수 |
| `--ewc_lambda` | `0.0` | EWC 강도 (`0` 이면 비활성, 권장 시작값 `500`) |
| `--category_weights` | `None` | 카테고리별 가중치 JSON 문자열 (예: `'{"emotion": 2.0, "long_dialogue": 1.5}'`) |

## 5. 과적합 분석 및 대응

### 학습 루프의 과적합 유발 요소

| 요소 | 현재 기본값 | 영향 | 권장 |
|---|---|---|---|
| `weight_decay` | `0.0` | L2 정규화 없음 → 가중치 무제한 성장 | `0.01` |
| `lora_dropout` | `0.05` | 소규모 데이터에서 너무 낮음 | `0.1` |
| `labels = _mask_labels` | assistant 토큰만 loss (v7~) | 전체 토큰보다 loss 값 자체가 높게 나옴 — v6 이전과 직접 비교 불가 | ✅ v7 적용 완료 |
| 샘플 수 3,170건 | 고정 (v11) | 적은 데이터 × 많은 epoch = 빠른 암기 | `--max_samples` 로 조절 |

epoch  5   eval 0.79    ↑ 과적합 시작
epoch 30   train 0.07 / eval 1.02  ↑ 완전 암기
```

```

epoch 10   train 0.08  ↑ 과적합 진행
```

```
epoch ~1.5  eval 1.714  ← 초기 수렴
epoch ~2.2  eval 1.760  ↑ 반등 시작 (과적합 조기 진입)

### 샘플링 동작 방식

`--max_samples N` 지정 시 **파일별 비율 유지 (stratified)** 방식으로 추출.

```
전체 데이터 로드 (57개 파일, 3,170건)  ← v11 기준; v10까지는 2,150건
    │
    ▼  파일별 quota = ceil(N × 파일건수 / 전체건수)
    │
    ▼  각 파일 내 shuffle(seed=42).select(quota)
    │
    ▼  전체 합산 shuffle(seed=42)
    │
    ├─ 90%  train_ds
    └─ 10%  eval_ds
```

> 파일별 비율이 유지되므로 특정 카테고리가 과소/과대 대표되는 문제 없음.

### 7-1. Assistant 토큰 마스킹 (Loss 범위 개선)

**현재 문제:**
`labels = input_ids`로 설정되어 있어 시스템 프롬프트·유저 턴 토큰에도 loss가 계산됨.
모델이 "응답"뿐 아니라 "입력받는 방식"까지 암기하게 되어 과적합을 가속시킴.

**개선 방향:**
assistant 응답 구간 토큰만 labels를 살리고, 나머지는 `-100`으로 마스킹.

```
현재:  [<system>...][<user>...][<assistant>그냥]
loss:       ✓              ✓           ✓

개선 후: [<system>...][<user>...][<assistant>그냥]
loss:      -100          -100          ✓
```

기대 효과: 과적합 억제 + 응답 품질 직접 최적화 + eval loss 곡선 안정화

---

### 7-2. EWC 기반 다단계 연속 학습 ✅ 구현 완료 (`training/ewc.py`)

**현재 문제:**
매 버전을 처음부터 학습 — 이전 버전이 잘 배운 패턴도 새 데이터에 의해 덮어쓰여질 수 있음.
특히 log 데이터 추가 학습 시 기존 speech_style 패턴이 흐트러질 위험이 있음.

**개선 방향:**
EWC(Elastic Weight Consolidation)를 다단계 학습에 적용.

```
[vN 학습 완료]
    ↓
Fisher 계산  →  output/LoRA_vN/fisher.pt
                output/LoRA_vN/ref_params.pt
    ↓
eval 테스트  →  카테고리별 약점 파악
    ↓
[vN+1 학습]
    L_total = L_CE + λ/2 × Σ F_i × (θ_i - θ*_i)²
              ↑ 새 데이터    ↑ 이전 버전 보호 패널티
```

**Fisher Information:**
LoRA 파라미터(~30M)에 대해서만 `grad²` 평균으로 추정.
중요도 높은 파라미터(= 이전 학습에서 자주 업데이트된 파라미터)일수록 새 학습에서 더 강하게 고정.

**λ 튜닝 기준:**

| λ | 효과 |
|---|---|
| 0.1~1.0 | 효과 미미 — EWC 패널티가 학습 loss에 묻힘 |
| **500** | **권장 시작값 (v9 실험 기준)** |
| 1000+ | 강한 보호, 새 적응 제한 — 특정 카테고리만 추가할 때 |

---

### 7-3. 카테고리 가중치 샘플링 ✅ 구현 완료 (`training/dataset.py category_weights`)

**현재 문제:**
파일별 비율 고정 stratified sampling — 테스트에서 약한 카테고리를 발견해도 해당 부분만 집중 학습 불가.

**개선 방향:**
JSONL 레코드의 `category` 필드 기준으로 카테고리별 오버샘플링 배율 지정.

```
--category_weights '{"affection": 2.0, "memory_ref": 1.5}'

동작:
  affection 카테고리 → 전체 샘플의 2배 비율로 추출 (반복 허용)
  memory_ref 카테고리 → 1.5배 비율로 추출
  나머지 카테고리 → 기존 비율 유지
```

EWC와 조합 시 활용법:
- EWC로 이전 강점 보호하면서 약한 카테고리만 집중 학습
- λ 높게 설정 + category_weights 높게 설정 → 특정 카테고리 핀포인트 개선

---

### 7-4. KL Divergence 규제 (미래 후보)

> 현재 보류. DPO 파이프라인 구축 이후 검토.

- `L_total = L_CE + β × KL(π_new || π_ref)`
- EWC와 달리 파라미터 공간이 아닌 **출력 분포 공간**에서 규제
- 학습 중 reference 모델의 forward pass가 추가로 필요 → VRAM 증가
- RLHF/DPO 방향으로 확장할 때 적합

---

### 7-5. VRAM 절감 기법 (현재 환경 기준)

> 현재 환경: RTX 5060 Ti VRAM 8GB / BitsAndBytes 4-bit 미지원(Blackwell SM 10.x) / 학습 중 VRAM 점유 ~96%.
> BnB 4-bit 모델 로드는 불가능하지만, **옵티마이저 상태**와 **어텐션 연산** 쪽에서 별도로 절감 가능.

#### ① 8-bit AdamW (옵티마이저 상태 절감)

```
현재 bfloat16 AdamW 옵티마이저 상태:
  파라미터 1개당 → m(1st moment) + v(2nd moment) = 2 × float32 = 8 bytes
  LoRA 30M 파라미터 기준 → 약 240 MB

8-bit AdamW 적용 시:
  m, v를 8-bit로 압축 → float32 대비 약 75% 절감 → 약 60 MB
```

bitsandbytes의 **4-bit 모델 로드**는 Blackwell 미지원이지만,
**옵티마이저 8-bit**는 모델 가중치 양자화가 아니므로 별도 동작 가능성 있음.

```python
# TrainingArguments에서 optim 변경만으로 적용
optim="adamw_bnb_8bit"   # 현재: "adamw_torch"
# 또는
optim="paged_adamw_8bit" # VRAM 부족 시 optimizer state를 CPU RAM으로 page-out
```

> ⚠️ Blackwell에서 bnb optimizer도 동작 여부 사전 확인 필요.
> `paged_adamw_8bit`는 CPU RAM(16GB 기준 여유 있음)으로 overflow → 안전한 폴백.

#### ② Flash Attention 2 (어텐션 연산 절감)

기존 어텐션은 시퀀스 길이 N에 대해 N² 크기의 attention matrix를 VRAM에 유지.
Flash Attention 2는 이를 **블록 단위(block-wise)로 타일링**하여 VRAM에 전체를 올리지 않고 처리.

```
기존 Attention:
  Q·K^T 행렬 (N × N) → VRAM에 상주
  N=512, bfloat16 기준 → 약 512KB / 레이어 → 28레이어 × ~14 MB

Flash Attention 2:
  블록 크기(BLOCK_M, BLOCK_N)로 나눠 SRAM에서 처리
  → attention matrix 전체를 VRAM에 올리지 않음 → ~30-40% 절감
```

```python
# AutoModelForCausalLM.from_pretrained에 추가
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    dtype=torch.bfloat16,
    attn_implementation="flash_attention_2",  # 추가
    ...
)
```

```bash
pip install flash-attn --no-build-isolation
```

> Qwen2.5는 Flash Attention 2 공식 지원. Blackwell CUDA 12.8 환경에서 동작 확인 필요.
> 적용 시 추가 설치 없이 `attn_implementation` 파라미터 하나로 전환 가능 — 우선순위 ①.

#### ③ GaLore (Gradient Low-Rank Projection)

LoRA는 **가중치**에 저랭크 행렬을 추가하는 방식. GaLore는 **그래디언트**를 저랭크 부분공간에 투영.

```
LoRA 방식:
  W_eff = W_frozen + B·A   ← A, B 파라미터 VRAM 점유
  전체 파라미터 그래디언트 계산 → LoRA 파라미터만 업데이트

GaLore 방식:
  W_full 전체 그래디언트 계산
  → SVD로 저랭크 부분공간 추출 (주기적 갱신, 예: 200 step마다)
  → 압축된 그래디언트만 옵티마이저에 전달
  → 추가 파라미터 없음, 기존 가중치 직접 업데이트
```

| 항목 | LoRA (rank=16) | GaLore (rank=16) |
|---|---|---|
| 추가 파라미터 | ~30M (VRAM) | 0 (VRAM 없음) |
| 옵티마이저 상태 | LoRA 파라미터 기준 | 투영된 gradient 기준 (더 작음) |
| 표현력 | 제한적 (삽입 위치 고정) | 전체 가중치 업데이트 가능 |
| 구현 난이도 | 낮음 (PEFT 라이브러리) | 중간 (별도 설치 필요) |

```bash
pip install galore-torch
```

> 소규모 SFT 파인튜닝에서 LoRA 대비 동등하거나 더 나은 품질로 VRAM 절감.
> 단, step마다 SVD 계산 비용이 있어 학습 속도가 소폭 저하될 수 있음.

### 7-6. 스케줄러 전략

현재 **Cosine Decay + 5% warmup** 사용. 대안 비교:

| 스케줄러 | 특징 | 적합한 상황 |
|---|---|---|
| **cosine** (현재) | 부드러운 감쇠, 안정적 | 일반 SFT |
| **WSD** (Warmup-Stable-Decay) | warmup → 일정 lr 유지 → 선형 감쇠 | 도메인 적응, 카테고리 균형 학습 |
| **linear** | 선형 감쇠, 단순 | 데이터 적고 epoch 적을 때 |
| **constant_with_warmup** | warmup 후 lr 고정 | 짧은 테스트 런 / step 수 불확실할 때 |
| **inverse_sqrt** | 초반 빠른 하강 후 완만 | 대용량 데이터, 장시간 학습 |

**WSD (Warmup-Stable-Decay) 권장 이유:**

```
cosine:
  lr │  ╮          ← warmup 후 바로 감쇠 시작
     │   ╰─────╮
     └──────────── step

WSD:
  lr │  ╮──────────╮   ← stable 구간에서 카테고리 전체를 고르게 학습
     │              ╲
     └──────────────── step
```

도메인 특화 SFT(캐릭터 말투, 감정 반응 등 카테고리가 다양할 때)에서
stable 구간이 충분히 확보되면 **모든 카테고리를 고르게 수렴**시키는 효과.

```python
# TrainingArguments
lr_scheduler_type="warmup_stable_decay",  # transformers 4.46+ 지원
warmup_ratio=0.05,
# stable 구간 끝 비율은 warmup_stable_decay_ratio 파라미터로 조정 (기본 0.9)
```

> transformers 버전이 낮으면 미지원. `constant_with_warmup` + 수동 step 지정으로 근사 가능.

### 7-7. 데이터 구성 전략 — 실측 기반 권장 설정

`--max_samples -1` + `training/data` 카테고리 비율 그대로 사용 시 가장 안정적인 대화 품질이 확인됨.

**이유:**

```
training/data/ 카테고리 구성
  affection/   (9파일)  ← 친밀도 단계별 반응 — 양이 가장 많음
  personality/ (5파일)  ← 성격 유형
  speech_style/(7파일)  ← 말투 조합
  common/      (3파일)  ← memory_ref / ai_tell_removal / persona_follow

stratified sampling이 이 비율을 유지 → 어느 한 카테고리가 과대표되지 않음
→ affection(친밀도) 변화에 따른 말투 변화가 자연스럽게 학습됨
```

**권장 설정 조합:**

```bash
--max_samples -1      # 전체 2,401건 사용 (emotion + long_dialogue 추가 후)
--data_dir training/data  # 카테고리 구조 그대로
--epochs 3            # v8 실측: 5 epoch에서 과적합. 3 권장
--eval_split 0.1      # best checkpoint 자동 선택 (VRAM 여유 있을 때)
--weight_decay 0.01   # 과적합 억제
--lora_dropout 0.1    # 소규모 데이터 기준
```

> `--max_samples` 를 1,200처럼 줄이면 유효 토큰 수 감소 + 조기 암기 → v7에서 실측 확인.
> 전체 데이터(-1)를 카테고리 비율 유지로 쓰는 것이 현 환경에서 최적.

### 7-8. 캐릭터 챗봇 파인튜닝 — 실무 접근법

동일 과제(특정 페르소나/말투/감정 반응 특화 소규모 SFT)에서 실무에서 자주 선택하는 방식:

#### ① SFT → DPO 2단계 (가장 일반적)

```
[1단계] SFT (현재 방식)
  고품질 데이터 → 기본 말투/페르소나/감정 반응 주입
  목표: loss 수렴, 캐릭터 붕괴 없음

[2단계] DPO (Direct Preference Optimization)
  (chosen, rejected) 쌍 데이터 준비
    chosen: 캐릭터에 맞는 이상적 응답
    rejected: AI투 표현 / 캐릭터 이탈 응답
  → 선호/거부 학습으로 미세 조정
  → reference 모델(SFT 완료본)과 비교하며 β × KL 규제
```

DPO는 RLHF 없이 preference 학습 가능. SFT 어댑터 기반으로 추가 학습이라 VRAM 부담 낮음.

#### ② ORPO (SFT + 선호 학습 1단계 통합)

```python
# trl 라이브러리
from trl import ORPOTrainer

# SFT loss + odds ratio preference loss를 동시에 계산
# reference 모델 forward pass 불필요 → DPO 대비 VRAM ~절반
# 단, 데이터 형식이 (prompt, chosen, rejected) 3항으로 구성돼야 함
```

DPO보다 심플하고 VRAM 효율적. 소규모 단일 GPU 환경에 적합.
캐릭터 챗봇 과제에서 "AI투 표현 줄이기" 같은 명확한 거부 쌍이 있을 때 효과적.

#### ③ 데이터 품질 원칙

실무에서 캐릭터 SFT에 공통으로 적용하는 원칙:

| 원칙 | 설명 |
|---|---|
| **고품질 > 고수량** | 정제된 500건 > 반복·노이즈 많은 5,000건 |
| **카테고리 균형** | 특정 패턴 과대표 시 모델이 해당 패턴만 반복 → stratified 필수 |
| **부정 예시 포함** | AI투 표현 제거 데이터(ai_tell_removal.jsonl)가 positive만큼 중요 |
| **말투 일관성** | 동일 캐릭터 내 speech_style 혼재 시 붕괴 발생 — 캐릭터별 분리 학습 권장 |
| **짧은 응답 다양화** | 단답형 응답이 70%+ → 길이 다양성 부족 시 모델이 단답만 생성 |

#### ④ 단계별 학습 (Curriculum Learning 변형)

```
[1라운드] 일반 말투 + 기본 성격 데이터 (speech_style/ + personality/)
    epoch 3~4 → best 수렴 확인
    ↓
[2라운드] 친밀도 변화 + 감정 반응 데이터 (affection/ + common/)
    EWC로 1라운드 패턴 보호하면서 추가 학습
    ↓
[3라운드] 실대화 로그 (training/log/ 수집 데이터)
    실사용 분포에 맞게 미세 보정
```

각 라운드에서 eval 결과를 보고 다음 라운드 데이터 선택 — 7-2(EWC) + 7-3(카테고리 가중치)와 조합.

> 현재 구조(training/data 카테고리 분리)가 이 방식을 바로 적용할 수 있는 형태로 이미 갖춰져 있음.

### 7-8-E. LoRA 학습 대상 영역 원칙

모든 기능에 LoRA를 적용하는 것은 학습 비용 대비 효율이 낮다.
아래 원칙에 따라 LoRA 투자 영역을 결정한다.

| 영역 | LoRA 필요성 | 이유 |
|---|---|---|
| **대화 (캐릭터 말투)** | **높음** | 반말 일관성·단답형·감정 억제는 프롬프트만으로 유지 어려움 |
| **나레이션** | 중간 | few-shot 프롬프트로 1차 제어 가능. 품질 부족 확인 시 추가 고려 |
| **기능 모드 파라미터 추출** | **낮음** | greedy decoding + repetition_penalty 조합으로 현재 충분 |
| **기능 결과 요약/응답** | **낮음** | Layer F 프롬프트 주입으로 대응. 3B 한국어 요약은 프롬프트 튜닝으로 보완 |
| **prompt_convert 변환** | **낮음** | 이미지 모델 특화 지식은 크롤링+프롬프트 파이프라인이 더 효과적 |

**프롬프트 엔지니어링 우선 적용 원칙:**

LoRA를 추가하기 전 반드시 프롬프트 단계에서 먼저 시도해야 한다.

```
[품질 개선 우선순위]
1. 프롬프트 구조 개선 (system_prompt 내용 정교화)
2. few-shot 예시 삽입 (나레이션·요약 등 문체 제어)
3. 위 두 단계로 부족할 때 → LoRA 학습 데이터 수집 및 추가 학습
```

이 원칙은 배포 환경(llama_cpp)에서 런타임 어댑터 교체가 불가능한 제약과도 연결된다.
프롬프트 방식은 전환 비용 0, 모델 재로딩 없음.

> 상세 배경: [docs/BUG/BUG_03.md](../docs/BUG/BUG_03.md) — 배포 환경 LoRA on/off 한계 및 선택지

### 7-9. 학습 모니터링 및 조기 종료 (`train_monitor.py`)

#### 개요

`training/train_monitor.py` 는 학습 명령어를 **그대로 입력** 받아 학습 프로세스를 subprocess로 띄우고,
`trainer_state.json` 을 주기적으로 읽어 과적합을 감지하면 자동으로 학습을 멈추는 래퍼 스크립트다.

```
[사용자]
    ↓  python training/train_monitor.py -- python -m training.lora_train ...
[train_monitor.py]
    ├─ subprocess.Popen → lora_train.py 실행
    ├─ 15초마다 trainer_state.json 폴링
    │     eval_loss / train_loss 파싱
    │     과적합 조건 검사
    │       ① eval_loss N회 연속 상승
    │       ② train/eval gap > 임계값
    ├─ 조건 충족 시 SIGTERM 전송 → 프로세스 정상 종료 대기
    ├─ VRAM 정리 (torch.cuda.empty_cache + synchronize)
    └─ 최종 보고 출력 (①학습상황 ②종료조건 ③입력명령어)
```

> **왜 subprocess인가?**
> SIGTERM 전송 시 HuggingFace Trainer 는 현재 체크포인트를 그대로 유지하고 종료.
> 별도 프로세스이므로 CUDA context 가 완전히 분리 → VRAM 해제 신뢰성 높음.

#### 사용법

```bash
# 기본 (-- 뒤에 학습 명령어 전달)
python training/train_monitor.py -- \
    python -m training.lora_train \
    --output_dir output/LoRA_v8 \
    --epochs 5 \
    --eval_split 0.1 \
    --weight_decay 0.01

# --cmd 로 문자열 전달
python training/train_monitor.py \
    --cmd "python -m training.lora_train --output_dir output/LoRA_v8 --epochs 5"

# 종료 조건 직접 설정
python training/train_monitor.py \
    --n_rise 2 \
    --gap 0.8 \
    --poll 30 \
    -- python -m training.lora_train --output_dir output/LoRA_v8
```

#### 인자 목록

| 인자 | 기본값 | 설명 |
|---|---|---|
| `--n_rise` | `3` | eval_loss 연속 상승 허용 횟수 (초과 시 종료) |
| `--gap` | `1.2` | train/eval loss gap 임계값 (초과 시 즉시 종료) |
| `--poll` | `15` | trainer_state.json 폴링 간격 (초) |
| `--cmd` | `None` | 학습 명령어 문자열 (`--` 방식과 택일) |

#### 종료 조건 상세

```
조건 ①: eval_loss 연속 상승
  eval_losses[-4:] = [1.71, 1.74, 1.78, 1.82]  → 3회 연속 상승 → 종료
  best=1.71 → 현재=1.82 로 보고

조건 ②: train/eval loss gap 초과
  train=0.35, eval=1.72  → gap=1.37 > 1.2  → 즉시 종료
```

> `--eval_split 0` (eval 없음) 이면 `trainer_state.json` 에 eval_loss 가 없으므로 두 조건 모두 발동 안 됨.
> 모니터가 의미 있으려면 **반드시 `--eval_split 0.1` 이상** 설정.

#### 최종 출력 예시

```
==============================================================
[ 모니터 최종 보고 ]

① 학습 상황
   eval 체크포인트 수 : 12
   best eval_loss     : 1.6870  (step 800)
   최종 eval_loss     : 1.8220
   최종 train_loss    : 0.3620

② 종료 조건: eval_loss 3회 연속 상승 (best=1.6870 → 현재=1.8220)

③ 입력 명령어: python -m training.lora_train --output_dir output/LoRA_v8 --epochs 5 ...
==============================================================
```

#### 권장 조합

```bash
python training/train_monitor.py \
    --n_rise 3 \
    --gap 1.2 \
    -- \
    python -m training.lora_train \
    --output_dir output/LoRA_v8 \
    --max_samples -1 \
    --epochs 5 \
    --eval_split 0.1 \
    --weight_decay 0.01 \
    --lora_dropout 0.1
```

`--n_rise 3` + `--gap 1.2` 는 v7 실측치(best step 800, gap 1.33)를 기준으로 설정한 값.
데이터 규모나 에폭 수에 따라 조정 — epoch 수가 많을수록 `--n_rise` 를 낮게 잡는 것이 안전.
