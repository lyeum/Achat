# LoRA 학습 매뉴얼

> 환경: Ubuntu 24.04 WSL2 / RTX 5060 Ti (VRAM 8GB / RAM 8GB) / CUDA 12.8
> 모델: Qwen2.5-3B-Instruct
> 방식: bfloat16 풀 파라미터 + LoRA (BitsAndBytes 4-bit 미사용 — Blackwell SM 10.x 미지원)

---

## 0. 현재 학습 방식 구조

### 0-1. LoRA가 가중치를 다루는 방식

기존 베이스 모델(Qwen2.5-3B)의 **모든 파라미터는 완전히 동결(freeze)** 된다.
새로운 가중치를 직접 추가하는 대신, 각 대상 레이어에 **저랭크 행렬 쌍 A·B를 삽입**하여 그것만 학습한다.

```
[기존 레이어 — 완전 동결]
    W_original (예: q_proj, 2048×2048)
    requires_grad = False  ← 절대 바뀌지 않음

[LoRA 삽입 — 이것만 학습]
    W_A  shape: (r, in_dim)    r=16 이면 (16, 2048)
    W_B  shape: (out_dim, r)   r=16 이면 (2048, 16)
    requires_grad = True

[추론 시 실제 적용값]
    W_effective = W_original + (alpha/r) × (W_B @ W_A)
                              ↑ 스케일링: alpha=32, r=16 → ×2.0
```

`W_A`는 랜덤 가우시안으로 초기화, `W_B`는 0으로 초기화 → **학습 시작 시 LoRA 기여분은 0**이라 안정적.

---

### 0-2. 현재 학습 대상 레이어 (target_modules)

Transformer 블록의 Attention 4개 + FFN 3개, **전 28개 레이어에 걸쳐** 모두 적용.

| 모듈 | 역할 | 위치 |
|---|---|---|
| `q_proj` | Query 투영 | Attention |
| `k_proj` | Key 투영 | Attention |
| `v_proj` | Value 투영 | Attention |
| `o_proj` | Output 투영 | Attention |
| `gate_proj` | FFN 게이트 (SiLU) | Feed-Forward |
| `up_proj` | FFN 업 투영 | Feed-Forward |
| `down_proj` | FFN 다운 투영 | Feed-Forward |

Bias는 모두 동결(`bias="none"`). Embedding / LayerNorm 도 동결.

---

### 0-3. 파라미터 수

```
전체 모델:    3,090,000,000 파라미터  (동결)
LoRA 추가분:     ~30,000,000 파라미터  (학습)  ← 전체의 약 1%

계산:
  레이어당 LoRA 파라미터 = 7개 모듈 × 2 × (r × dim)
  예) r=16, dim=2048 → 7 × 2 × 16 × 2048 ≈ 458,752
  28 레이어 × 458,752 ≈ 12.8M  (단방향 기준 — 실제는 양방향 포함 ~30M)
```

### 0-4. 정밀도 및 메모리 관리

| 항목 | 설정 | 이유 |
|---|---|---|
| 모델 로드 정밀도 | **bfloat16** | RTX 5060 (Blackwell) 4-bit 양자화 미지원 |
| LoRA 가중치 정밀도 | bfloat16 (자동) | 모델과 동일 dtype 따라감 |
| Gradient 계산 | float32 누적 | 기울기 소실 방지 |
| Gradient Checkpointing | **활성** | Activation을 저장 않고 backward 시 재계산 → VRAM ~30% 절약 |
| `use_cache = False` | 강제 비활성 | KV 캐시와 gradient checkpointing 충돌 방지 |

### 0-5. Loss 계산 범위 (현재 방식의 한계)

```
[토크나이징 결과]
input_ids: [<system>...하루다</s>][<user>뭐해</s>][<assistant>그냥</s>]
labels:    [  동일  ............][  동일  .......][   동일  ........]
                ↑                      ↑                  ↑
            loss 계산됨           loss 계산됨         loss 계산됨
```

현재는 `labels = input_ids` 로 **전체 토큰에 대해 loss를 계산**한다.
시스템 프롬프트·유저 턴까지 loss에 포함되므로 모델이 "무엇을 예측해야 하는가"가 다소 모호해진다.

> **v7부터 적용**: assistant 토큰 구간만 loss 계산 (시스템·유저 구간 labels → -100 마스킹)
> → 응답 품질 직접 최적화, 과적합 억제 효과. `tokenize_dataset._mask_labels()` 구현 완료.

---

### 0-6. 옵티마이저 및 스케줄러

```
옵티마이저: AdamW
  - lr: 2e-4 (초기값)
  - weight_decay: 0.01 (L2 정규화)
  - beta1=0.9, beta2=0.999 (기본값)

스케줄러: Cosine Decay
  - warmup: 전체 스텝의 5% (loss 안정화 후 감쇠 시작)
  - 최종 lr → 0 으로 수렴

유효 배치:
  batch_size(1) × grad_accum(8) = 8
  → 8샘플 분의 gradient를 누적 후 1회 업데이트
```

---

### 0-7. Best Model 선택 방식

```
eval_split > 0 일 때:
  save_steps(100)마다 eval loss 계산
  → 가장 낮은 eval loss 시점의 가중치를 자동 선택
  → 학습 완료 후 adapter/ 에 best 가중치 저장

eval_split = 0 일 때:
  best 자동 선택 불가 → 마지막 epoch 가중치 저장
  (v3 실측: epoch 3~4가 best, 이후 eval loss 반등)
```

## 1. 학습 루프 내부 흐름

```
[데이터 로드]
training/data/**/*.jsonl (또는 data/lora/)
    │
    ▼ load_training_data()
HuggingFace Dataset 객체 (text 필드)
    │
    ▼ tokenize_dataset()   — truncation=True, max_length=512
    │                        labels = _mask_labels(input_ids)
    │                        → assistant 응답 구간만 loss 계산 (v7~)
    │                        → 시스템·유저 토큰 구간은 labels=-100 마스킹
    │
    ▼ train_test_split()   — eval_split > 0 이면 90% train / 10% eval 분리
    │                        eval_split = 0 이면 전체 train, eval 없음
    │
[모델 로드]
AutoModelForCausalLM (bfloat16, device_map="auto")
    │
    ▼ get_peft_model()     — LoRA 어댑터 주입
    │                        target: q/k/v/o/gate/up/down proj (전 레이어)
    │                        trainable params: ~30M / 전체 3.1B
    │
[학습 루프 — HuggingFace Trainer]
    │
    ├─ 매 스텝:
    │   forward → loss 계산 → backward → gradient accumulation(8스텝)
    │   → optimizer step (AdamW + cosine LR decay)
    │
    ├─ logging_steps마다:  train loss / grad_norm / lr / epoch 출력
    │
    ├─ save_steps(100)마다:
    │   eval_split > 0 → eval loss 계산 → best이면 checkpoint 교체
    │   eval_split = 0 → checkpoint만 저장 (eval 없음)
    │
    └─ 학습 완료:
        load_best_model_at_end=True (eval 있을 때) → best checkpoint 복원
        model.save_pretrained() → output/{run_name}/adapter/ 저장
```

**총 스텝 수 계산:**
```
총 스텝 = (샘플 수 × 0.9) ÷ (batch_size × grad_accum) × epochs
예 | 2167 × 0.9 ÷ (1 × 8) × 4 ≈ 975 스텝  (v5 기준 — 2,167건)
```

## 5. 과적합 분석 및 대응

### 학습 루프의 과적합 유발 요소

| 요소 | 현재 기본값 | 영향 | 권장 |
|---|---|---|---|
| `weight_decay` | `0.0` | L2 정규화 없음 → 가중치 무제한 성장 | `0.01` |
| `lora_dropout` | `0.05` | 소규모 데이터에서 너무 낮음 | `0.1` |
| `labels = _mask_labels` | assistant 토큰만 loss (v7~) | 전체 토큰보다 loss 값 자체가 높게 나옴 — v6 이전과 직접 비교 불가 | ✅ v7 적용 완료 |
| 샘플 수 2,167건 | 고정 | 적은 데이터 × 많은 epoch = 빠른 암기 | `--max_samples` 로 조절 |

> ⚠️ **v7부터 loss 스케일이 달라짐**: assistant 토큰만 계산하므로 같은 품질이어도 v7 loss 값이 v1~v6보다 높게 나온다.
> v7 이후 버전끼리만 loss 값으로 비교할 것.

**실제 관측 (lora_haru_v3, 30 epoch) — 전체 토큰 loss 기준:**
```
epoch  3   eval 0.7226  ← best
epoch  5   eval 0.79    ↑ 과적합 시작
epoch 30   train 0.07 / eval 1.02  ↑ 완전 암기
```

**실제 관측 (LoRA_v6, 10 epoch) — 전체 토큰 loss 기준:**
```
epoch ~5   eval 0.7198  ← best  (v3 대비 개선)
epoch 10   train 0.08  ↑ 과적합 진행
```

**실제 관측 (LoRA_v7, 4 epoch, max_samples=1200) — assistant 토큰 loss 기준 ⚠️:**
```
epoch ~1.5  eval 1.714  ← 초기 수렴
epoch ~2.2  eval 1.760  ↑ 반등 시작 (과적합 조기 진입)
epoch ~3    eval 1.687  ← best
epoch ~3.7  eval 1.822  ↑ 과적합 진행
train  4    loss 0.362    (train/eval gap 1.33 — 과적합 심함)
```
→ 원인: max_samples=1200 (train 1,080건) + assistant 마스킹으로 샘플당 유효 토큰 감소 → 조기 암기
→ **v8 대응: max_samples=-1 (전체 2,167건), 5 epoch**

### 샘플링 동작 방식

`--max_samples N` 지정 시 **파일별 비율 유지 (stratified)** 방식으로 추출.

```
전체 데이터 로드 (26개 파일, 2,167건)
    │
    ▼  파일별 quota = ceil(N × 파일건수 / 전체건수)
    │
    ▼  각 파일 내 shuffle(seed=42).select(quota)
    │
    ▼  전체 합산 shuffle(seed=42)
    │
    ├─ 90%  train_ds
    └─ 10%  eval_ds
```

> 파일별 비율이 유지되므로 특정 카테고리가 과소/과대 대표되는 문제 없음.

## 7. 학습 개선안

> 구현 계획: [docs/plan/training_개선.md](../docs/plan/training_개선.md)

### 7-1. Assistant 토큰 마스킹 (Loss 범위 개선)

**현재 문제:**
`labels = input_ids`로 설정되어 있어 시스템 프롬프트·유저 턴 토큰에도 loss가 계산됨.
모델이 "응답"뿐 아니라 "입력받는 방식"까지 암기하게 되어 과적합을 가속시킴.

**개선 방향:**
assistant 응답 구간 토큰만 labels를 살리고, 나머지는 `-100`으로 마스킹.

```
현재:  [<system>...][<user>...][<assistant>그냥]
loss:       ✓              ✓           ✓

개선 후: [<system>...][<user>...][<assistant>그냥]
loss:      -100          -100          ✓
```

기대 효과: 과적합 억제 + 응답 품질 직접 최적화 + eval loss 곡선 안정화

---

### 7-2. EWC 기반 다단계 연속 학습

**현재 문제:**
매 버전을 처음부터 학습 — 이전 버전이 잘 배운 패턴도 새 데이터에 의해 덮어쓰여질 수 있음.
특히 log 데이터 추가 학습 시 기존 speech_style 패턴이 흐트러질 위험이 있음.

**개선 방향:**
EWC(Elastic Weight Consolidation)를 다단계 학습에 적용.

```
[vN 학습 완료]
    ↓
Fisher 계산  →  output/LoRA_vN/fisher.pt
                output/LoRA_vN/ref_params.pt
    ↓
eval 테스트  →  카테고리별 약점 파악
    ↓
[vN+1 학습]
    L_total = L_CE + λ/2 × Σ F_i × (θ_i - θ*_i)²
              ↑ 새 데이터    ↑ 이전 버전 보호 패널티
```

**Fisher Information:**
LoRA 파라미터(~30M)에 대해서만 `grad²` 평균으로 추정.
중요도 높은 파라미터(= 이전 학습에서 자주 업데이트된 파라미터)일수록 새 학습에서 더 강하게 고정.

**λ 튜닝 기준:**

| λ | 효과 |
|---|---|
| 0.1 | 약한 보호, 새 데이터 우선 |
| **0.5** | **권장 시작값** |
| 1.0 | 강한 보호, 새 적응 제한 |
| 5.0+ | 특정 카테고리만 추가할 때 |

---

### 7-3. 카테고리 가중치 샘플링

**현재 문제:**
파일별 비율 고정 stratified sampling — 테스트에서 약한 카테고리를 발견해도 해당 부분만 집중 학습 불가.

**개선 방향:**
JSONL 레코드의 `category` 필드 기준으로 카테고리별 오버샘플링 배율 지정.

```
--category_weights '{"affection": 2.0, "memory_ref": 1.5}'

동작:
  affection 카테고리 → 전체 샘플의 2배 비율로 추출 (반복 허용)
  memory_ref 카테고리 → 1.5배 비율로 추출
  나머지 카테고리 → 기존 비율 유지
```

EWC와 조합 시 활용법:
- EWC로 이전 강점 보호하면서 약한 카테고리만 집중 학습
- λ 높게 설정 + category_weights 높게 설정 → 특정 카테고리 핀포인트 개선

---

### 7-4. KL Divergence 규제 (미래 후보)

> 현재 보류. DPO 파이프라인 구축 이후 검토.

- `L_total = L_CE + β × KL(π_new || π_ref)`
- EWC와 달리 파라미터 공간이 아닌 **출력 분포 공간**에서 규제
- 학습 중 reference 모델의 forward pass가 추가로 필요 → VRAM 증가
- RLHF/DPO 방향으로 확장할 때 적합
